# 4. 카테고리 목록 페이지네이션

## 배경

- 현재 포스트 412개. `BlogListLayout.astro`는 카테고리의 모든 글을 한 페이지에 렌더링한다.
- Market Brief처럼 매일 쌓이는 카테고리는 목록 HTML이 계속 커진다 (로딩·스크롤 부담).
- 제약: 목록에는 **클라이언트 사이드 정렬 토글**(최신순/오래된 순, `sort-btn`)과 **pinned 섹션**(시작 가이드)이 있다. 서버 사이드 페이지 분할과 클라이언트 정렬은 충돌한다 — 페이지를 나누면 "오래된 순" 정렬이 현재 페이지 내에서만 뒤집힌다.

## 목표

글이 많은 카테고리에서 초기 렌더링 크기를 줄이되, 정렬 토글과 pinned 섹션의 동작을 유지한다.

## 설계 선택

두 가지 대안 중 **A안(클라이언트 사이드 "더 보기" / 점진 렌더링)** 을 권장한다.

### A안 — 클라이언트 사이드 점진 표시 (권장)

- 서버는 지금처럼 전체 목록을 렌더링하되, N개(예: 30) 이후 카드는 `hidden` 처리하고 "더 보기" 버튼으로 노출.
- 정렬 토글은 전체 DOM을 대상으로 동작하므로 기존 로직 유지 가능 (정렬 후 hidden 재적용).
- URL 구조 변경 없음 → SEO·기존 링크·RSS 영향 없음.
- 단점: HTML 자체 크기는 줄지 않음 (렌더링·이미지 로딩 부담만 감소, 카드에 이미지 lazy-loading 확인 필요).

### B안 — Astro `paginate()` 기반 정적 페이지 분할

- `/market-brief/2/` 같은 경로 생성. HTML 크기 자체가 줄어든다.
- 단점: 카테고리 index 페이지 8개를 모두 `[...page].astro` 구조로 재작성해야 하고, 정렬 토글을 "페이지 내 정렬"로 격하하거나 asc/desc 별도 경로를 만들어야 한다. pinned 섹션은 1페이지에만 표시하는 규칙 필요.

HTML 크기가 실제 문제로 확인되기 전까지는 A안으로 충분하다. 빌드 후 `dist/market-brief/index.html` 크기를 먼저 측정하고, 수백 KB를 넘으면 B안을 재검토한다.

## 변경 대상 (A안 기준)

1. `src/layouts/BlogListLayout.astro`
   - `regular-post-grid`의 카드에 index 기반 `data-index` 부여, 초기 N개 초과분에 hidden 클래스.
   - 목록 아래 "더 보기" 버튼 추가 (전부 표시되면 숨김).
   - 기존 정렬 스크립트에 "정렬 변경 시 표시 개수 유지" 로직 추가.
2. `src/components/BlogPostCard.astro` — 썸네일 이미지가 있다면 `loading="lazy"` 확인/추가.

## 구현 단계

1. `npm run build` 후 카테고리별 `dist/<category>/index.html` 크기 측정 → 기준선 기록.
2. `BlogListLayout.astro`에 표시 개수 상태(N=30)와 "더 보기" 버튼 구현.
3. 정렬 토글과의 상호작용 구현: 정렬 변경 시 현재 표시 개수를 유지한 채 순서만 변경.
4. pinned 섹션은 페이지네이션 대상에서 제외 (항상 전부 표시).

## 검증

1. `npm run check` + `npm run build`.
2. `npm run preview`로 Market Brief(글 최다 카테고리)에서:
   - 초기 30개 표시 → 더 보기 클릭 시 추가 표시.
   - 최신순/오래된 순 토글이 표시된 범위에 일관되게 동작.
   - pinned 섹션 정상 표시.
3. 모바일 뷰포트에서 버튼 터치 영역 확인 (AGENTS.md 리뷰 체크리스트).

## 리스크 / 참고

- JS 비활성 환경에서는 hidden 처리된 글이 안 보인다 → `<noscript>` 대응 또는 hidden을 JS로만 적용(초기 HTML은 전부 표시, JS 로드 후 접기)하는 방식으로 회피.
- 검색 인덱스(`search.json`)는 별개 경로이므로 영향 없음. 다만 같은 맥락에서 인덱스 크기(포스트당 최대 ~5.5KB 텍스트 × 412개 ≈ 2MB+)도 커지고 있으므로, `SEARCH_TEXT_LENGTH`(5000) 축소 또는 카테고리별 인덱스 활용을 별도 검토할 것.
