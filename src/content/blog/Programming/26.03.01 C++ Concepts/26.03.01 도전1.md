좋아요—“**콜백 정책 주입**”을 `std::predicate`/`std::invocable`로 깔끔하게 구현하는 패턴을 단계별로 보여드릴게요. 핵심 아이디어는 **범용 함수에 ‘판별 정책(predicate)’과 ‘동작 정책(action)’을 주입**해서, 타입 안정성과 가독성을 동시에 얻는 겁니다.

---

# 1) 단일 요소 처리: `process_if(range, pred, action)`

* `pred`는 요소를 받아 **bool로 판단**해야 하므로 `std::predicate`로 제약
* `action`은 요소를 받아 **무언가 수행**하면 되므로 `std::invocable`로 제약

```cpp
#include <concepts>
#include <ranges>
#include <utility>
#include <cstddef>

template <std::ranges::input_range R,
          typename Pred,
          typename Action>
requires std::predicate<Pred&, std::ranges::range_reference_t<R>> &&
         std::invocable<Action&, std::ranges::range_reference_t<R>>
std::size_t process_if(R&& r, Pred pred, Action action)
{
    std::size_t count = 0;
    for (auto&& e : r) {
        if (std::invoke(pred, e)) {         // std::predicate 제약으로 bool로 호출 가능 보장
            std::invoke(action, e);         // std::invocable 제약으로 호출 가능 보장
            ++count;
        }
    }
    return count;
}
```

### 사용 예

```cpp
#include <vector>
#include <string>
#include <iostream>

int main() {
    std::vector<int> v{1,2,3,4,5,6};

    // 짝수만 출력
    auto n1 = process_if(v,
        [](int x) { return x % 2 == 0; },        // predicate
        [](int x) { std::cout << x << ' '; }     // action
    );
    std::cout << "\nselected=" << n1 << "\n";

    // 길이가 3 이상인 문자열만 대문자로 로깅(상태를 가진 정책도 OK)
    struct Logger {
        int hits = 0;
        void operator()(const std::string& s) { std::cout << "[log] " << s << '\n'; ++hits; }
    } logger;

    std::vector<std::string> ss{"a","bbb","cc","dddd"};
    auto n2 = process_if(ss,
        [](const std::string& s){ return s.size() >= 3; },
        logger // stateful functor
    );
    std::cout << "hits=" << logger.hits << " selected=" << n2 << "\n";
}
```

**포인트**

* `std::predicate<Pred&, T>`는 `Pred`가 `T`를 인자로 **bool로 평가**된다는 것을 컴파일 타임에 보장합니다.
* `std::invocable<Action&, T>`는 `Action`이 `T`로 **호출 가능**하다는 것을 보장합니다.
* 런타임 분기 없이 **정적 타입 안정성** 확보!

# 2) 결과를 축적하는 정책: `filter_transform(range, pred, proj, sink)`

조금 더 심화해서, **선택된 요소**에 대해 **프로젝션(변환)**을 적용한 뒤, 결과를 **수집 정책**(예: `push_back`)으로 넘겨보죠.

```cpp
#include <vector>
#include <type_traits>

template <std::ranges::input_range R,
          typename Pred,
          typename Proj,
          typename Sink>
requires std::predicate<Pred&, std::ranges::range_reference_t<R>> &&
         std::invocable<Proj&, std::ranges::range_reference_t<R>> &&
         // Sink는 변환 결과로 호출 가능해야 함
         std::invocable<Sink&, std::invoke_result_t<Proj&, std::ranges::range_reference_t<R>>>
void filter_transform(R&& r, Pred pred, Proj proj, Sink sink)
{
    for (auto&& e : r) {
        if (std::invoke(pred, e)) {
            auto y = std::invoke(proj, e);
            std::invoke(sink, std::move(y));
        }
    }
}
```

### 사용 예

```cpp
#include <iostream>
#include <string>

int main() {
    std::vector<int> v{1,2,3,4,5,6};

    std::vector<double> out;
    out.reserve(v.size());

    filter_transform(v,
        [](int x){ return x % 2 == 0; },     // pred: 짝수만
        [](int x){ return x * 1.5; },        // proj: double로 스케일
        [&](double y){ out.push_back(y); }   // sink: 벡터에 적재
    );

    for (double d : out) std::cout << d << ' '; // 3 6 9
    std::cout << '\n';
}
```

**포인트**

* **정의역**(`R`의 요소) → **판별**(`Pred`) → **변환**(`Proj`) → **소비**(`Sink`)로 **정책 파이프라인** 구성.
* 각 정책은 개념 제약으로 **의도대로 연결**되는지만 검사하면 끝입니다.

# 3) 반환 타입을 제약하고 싶다면?

예를 들어 변환 결과가 **숫자여야** 한다면 `std::invocable`의 결과에 **추가 개념**을 걸 수 있습니다.

```cpp
template <typename T>
concept NumberLike = std::integral<T> || std::floating_point<T>;

template <std::ranges::input_range R, typename Pred, typename Proj, typename Sink>
requires std::predicate<Pred&, std::ranges::range_reference_t<R>> &&
         std::invocable<Proj&, std::ranges::range_reference_t<R>> &&
         NumberLike<std::remove_cvref_t<std::invoke_result_t<Proj&, std::ranges::range_reference_t<R>>>> &&
         std::invocable<Sink&, std::invoke_result_t<Proj&, std::ranges::range_reference_t<R>>>
void numeric_filter_transform(R&& r, Pred pred, Proj proj, Sink sink) {
    for (auto&& e : r) if (std::invoke(pred,e)) std::invoke(sink, std::invoke(proj, e));
}
```

# 4) Ranges와 함께 파이프라이닝(보너스)

정책을 **뷰**로 만들면 가볍게 연결할 수 있습니다. (아래는 `pred`만 뷰로, 나머지는 동작함수)

```cpp
#include <ranges>
#include <vector>
#include <iostream>

template <std::ranges::input_range R, typename Pred, typename Action>
requires std::predicate<Pred&, std::ranges::range_reference_t<R>> &&
         std::invocable<Action&, std::ranges::range_reference_t<R>>
void for_each_if(R&& r, Pred pred, Action action) {
    for (auto&& e : r | std::views::filter([&](auto&& x){ return std::invoke(pred, x); })) {
        std::invoke(action, e);
    }
}
```

# 5) 작은 `static_assert` 자기검증

```cpp
#include <type_traits>

static_assert(std::predicate<bool(*)(int), int>);
static_assert(!std::predicate<void(*)(int), int>); // void 반환 → predicate 아님

struct Fun { bool operator()(int) const; };
static_assert(std::predicate<Fun, int>);

struct Sink { void operator()(double) const; };
static_assert(std::invocable<Sink, double>);
static_assert(!std::invocable<Sink, int&>); // 시그니처가 안 맞으면 false
```

# 언제 `std::predicate` vs `std::invocable`?

* **조건 판단(불값)** 이 필요하면 → `std::predicate<P, Args...>` (반드시 `bool` 컨텍스트로 평가 가능해야 함)
* **그 외 임의 호출**이면 → `std::invocable<F, Args...>` (반환형 자유)

  * 필요하면 `std::regular_invocable`(동등성/연속 호출 안정성)이나 결과 타입을 `std::same_as`/`std::convertible_to`로 **추가 제약**하세요.

> `std::invocable`, `std::predicate` 추가 설명?

좋아요—`std::invocable`와 `std::predicate`를 “무엇/왜/어떻게” 관점에서 깔끔하게 정리해 드릴게요. 전부 **컴파일 타임 개념(Concepts)** 이라서, 함수 템플릿이나 API 앞에서 “콜백이 진짜 호출 가능한가요?” “불값 판별자 맞나요?”를 엄격하게 거를 때 씁니다.

---

# `std::invocable` — “호출만 되면 OK”

**정의(요지)**
`std::invocable<F, Args...>`는 **`std::invoke(std::declval<F>(), std::declval<Args>()...)` 가 잘 형성**되면 `true`입니다.
즉, **어떤 방식으로든 호출 가능**하면 통과합니다. (라이트한 제약)

* 대상: 함수 포인터, 함수 객체, 람다, 멤버 함수/데이터 포인터(수신자까지 포함) 등 **모든 호출 가능한 것**
* 반환 타입 조건 없음(무엇이든 가능, `void`도 OK)
* 상태 보존 여부, 재호출 가능성 등도 요구하지 않음

```cpp
#include <concepts>
#include <utility>

int f(int);              // 함수
struct G { int operator()(double) const; }; // 함수 객체
struct H { int mem(int) const; };

static_assert( std::invocable<decltype(f), int> );      // f(42)
static_assert( std::invocable<G, double> );             // G{}(3.14)
static_assert( !std::invocable<G, int> );               // 인자 타입 불일치
static_assert( std::invocable<int H::*, H&> == false ); // 데이터 멤버 포인터 자체는 호출 불가

// 멤버 함수 포인터는 '수신자'와 함께 호출 가능해야 invocable
static_assert( std::invocable<int (H::*)(int) const, const H&, int> );
```

> 요약: **“부를 수만 있으면”** 통과. 반환이 `bool`일 필요도 없습니다.

---

# `std::predicate` — “불값 판별자여야 OK”

**정의(요지)**
`std::predicate<P, Args...>`는 `P`가 **정상적으로 호출 가능**(사실상 `std::regular_invocable`)하면서, 그 **반환이 불리언 문맥에서 참/거짓으로 해석**될 수 있어야 합니다.
즉, “**조건을 판단하는 콜백**”임을 보장하는 개념이에요.

* `std::predicate`는 `std::invocable`보다 **강한 제약**입니다.
* 반환은 **`bool`이거나 `bool`로 변환 가능한 타입**이어야 함.
* Ranges/알고리즘에서 “필터/조건” 자리에 즐겨 씁니다.

```cpp
#include <concepts>

bool is_even(int);
int  nonzero(int);   // int는 bool로 변환 가능(0/!=0)

// 전형적인 판별자
static_assert( std::predicate<decltype(is_even), int> );

// int 반환도 '불리언 문맥'에는 OK
static_assert( std::predicate<decltype(nonzero), int> );

// void 반환은 판별자가 아님
void do_stuff(int);
static_assert( !std::predicate<decltype(do_stuff), int> );
```

> 요약: **“불값을 돌려주는 호출자”** 여야 통과. (호출 가능 + boolean-testable 반환)

---

# 둘의 관계 한 줄 요약

* `std::invocable<F, Args...>`: **호출 가능**이면 됨 → 가장 느슨한 제약
* `std::predicate<P, Args...>`: **호출 가능 + 반환이 참/거짓 문맥** → 필터/조건 전용

실전 감각으로는

* **액션/작업 콜백 자리**: `std::invocable`
* **필터/조건/비교 자리**: `std::predicate` (또는 `std::strict_weak_order` 같은 특화 개념)

# 실전 패턴 비교

## 1) 필터 콜백 (predicate)

```cpp
#include <ranges>
#include <vector>
#include <iostream>

template <std::ranges::input_range R, typename Pred>
requires std::predicate<Pred&, std::ranges::range_reference_t<R>>
void print_if(R&& r, Pred pred) {
    for (auto&& e : r)
        if (std::invoke(pred, e))
            std::cout << e << ' ';
}
```

## 2) 동작 콜백 (invocable)

```cpp
template <std::ranges::input_range R, typename Action>
requires std::invocable<Action&, std::ranges::range_reference_t<R>>
void for_each_do(R&& r, Action act) {
    for (auto&& e : r) std::invoke(act, e);
}
```

## 3) 둘 다 쓰기 (앞서 만든 “정책 주입”)

```cpp
template <std::ranges::input_range R, typename Pred, typename Action>
requires std::predicate<Pred&,   std::ranges::range_reference_t<R>> &&
         std::invocable<Action&, std::ranges::range_reference_t<R>>
void process_if(R&& r, Pred pred, Action act) {
    for (auto&& e : r)
        if (std::invoke(pred, e)) std::invoke(act, e);
}
```

# 자주 하는 질문(FAQ)

**Q. `std::predicate`는 꼭 `bool`을 반환해야 하나요?**
A. 아니요. **`bool`로 변환 가능**하면 됩니다. (`int`, 사용자 정의 타입이 `explicit operator bool()` 제공 등)

**Q. `std::regular_invocable`와 차이는?**
A. `regular_invocable`은 `invocable`을 다듬은 개념으로, **일반 함수 호출처럼** 동작할 것을 더 강하게 요구합니다(호출 문맥 일관성). `predicate`는 내부적으로 그 수준의 호출 가능성을 기대합니다. 보통 “조건 콜백이면 `predicate`”로 기억하시면 충분합니다.

**Q. 멤버 함수 포인터도 `invocable/predicate`로 검사 가능한가요?**
A. 네. 단, **수신자(객체/참조/포인터)** 를 함께 인자로 전달해야 `std::invoke`가 성립합니다.

```cpp
struct S { bool ok(int) const; };
using MF = bool (S::*)(int) const;

static_assert( std::predicate<MF, const S&, int> );    // S의 멤버 함수 포인터 + 수신자
static_assert( !std::predicate<MF, int> );             // 수신자 없음 → 호출 불가
```

# 손에 붙는 미니 치트시트

* **조건이면** `std::predicate<P, T...>`
* **그 외 호출이면** `std::invocable<F, T...>`
* 반환 타입을 추가로 제약하고 싶으면:

  * 정확히: `{ expr } -> std::same_as<bool>;`
  * 느슨히: `{ expr } -> std::convertible_to<bool>;`

```cpp
// 예: 'pred(e)'가 정확히 bool을 반환해야 한다면
template <typename Pred, typename T>
concept BoolPredicate =
    std::predicate<Pred&, T> &&
    requires (Pred p, T t) { { std::invoke(p, t) } -> std::same_as<bool>; };
```
