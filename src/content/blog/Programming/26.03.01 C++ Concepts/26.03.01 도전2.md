좋아요! “사용자 정의 타입이 `operator<<`를 제공하면 로그 API를 허용”하는 **커스텀 Concept** 패턴을 단계별로 보여드릴게요. 

핵심은 **Streamable(스트림 삽입 가능성)**을 개념으로 정의하고, 그걸로 **오버로드 선택**(또는 삭제 선언)으로 정책을 고르는 겁니다.


# 1) `StreamInsertable` / `Streamable` 컨셉 정의

`std::basic_ostream<CharT, Traits>& os; os << t` 가 **정상 표현**이고 **같은 스트림 타입을 반환**하면 통과하도록 만듭니다. (넓게 `wchar_t` 등까지 지원)

```cpp
// streamable.hpp
#include <concepts>
#include <ostream>
#include <string>
#include <ranges>

template <typename T, typename CharT = char, typename Traits = std::char_traits<CharT>>
concept StreamInsertable =
    requires (std::basic_ostream<CharT, Traits>& os, const T& t) {
        { os << t } -> std::same_as<std::basic_ostream<CharT, Traits>&>;
    };

// char 전용 축약 이름(가장 흔히 쓰는 std::ostream 기준)
template <typename T>
concept Streamable = StreamInsertable<T, char>;
```

> 왜 `std::ostream&`이 아니라 **`basic_ostream<CharT,Traits>&`** 로 검사?
> `std::wostream`, 커스텀 traits 등을 포괄하려고요. 좁혀도 되지만, 이렇게 작성하면 더 재사용성이 좋아요.
> 참고: 컨셉에서 **반환 타입**까지 명시하면(복합 요구) 진단이 깔끔해집니다.
> (C++ Core Guidelines / cppreference에서 `requires`-expression 참고)

# 2) `log` 오버로드: Streamable이면 바로 출력

```cpp
#include <iostream>

template <Streamable T>
void log(const T& x) {
    std::cout << x;
}

template <Streamable T>
void logln(const T& x) {
    std::cout << x << '\n';
}
```

* `Streamable`을 만족하는 타입만 이 오버로드가 **후보**가 됩니다.
* 만족 못 하면 **다른 오버로드**(아래)를 보거나, 없으면 컴파일 타임에 막습니다.

# 3) 비-Streamable을 위한 **대체 정책**(선택 사항)

## 3-1) `to_string()`이나 `std::to_string`을 허용하고 싶다면

```cpp
#include <type_traits>
#include <string>

template <typename T>
concept HasMemberToString =
    requires (const T& t) {
        { t.to_string() } -> std::convertible_to<std::string>;
    };

template <typename T>
concept StdToStringable =
    requires (const T& t) {
        { std::to_string(t) } -> std::convertible_to<std::string>;
    };

template <typename T>
concept ToStringLike = HasMemberToString<T> || StdToStringable<T>;

template <ToStringLike T>
void log(const T& x) {
    if constexpr (HasMemberToString<T>) std::cout << x.to_string();
    else                                std::cout << std::to_string(x);
}
```

## 3-2) Range의 원소가 Streamable이면 시퀀스 로깅

```cpp
template <std::ranges::input_range R>
requires Streamable<std::ranges::range_reference_t<R>>
void log_range(R&& r, std::string_view sep = ",") {
    bool first = true;
    for (auto&& e : r) {
        if (!first) std::cout << sep;
        std::cout << e;                 // 요소가 Streamable이라 안전
        first = false;
    }
}
```

## 3-3) **명시적으로 금지**(에러 메시지를 더 친절하게)

```cpp
template <typename T>
requires (!Streamable<T> && !ToStringLike<T>)
void log(const T&) = delete; // “이 타입은 로깅 정책을 만족하지 않습니다” 컴파일 오류
```

# 4) 사용자 정의 타입에서 `operator<<` 제공하기

### (A) 비멤버 연산자 (권장: ADL 잘 작동)

```cpp
#include <ostream>
struct User {
    int id; std::string name;
    friend std::ostream& operator<<(std::ostream& os, const User& u) {
        return os << "{id:" << u.id << ",name:" << u.name << "}";
    }
};
static_assert(Streamable<User>);
```

### (B) wide 스트림까지 의식한다면

```cpp
#include <ostream>
#include <string>
struct WideTitle {
    std::wstring title;
    friend std::wostream& operator<<(std::wostream& os, const WideTitle& w) {
        return os << w.title;
    }
};
static_assert(StreamInsertable<WideTitle, wchar_t>);
```

# 5) 사용 예 & `static_assert` 셀프 테스트

```cpp
struct Point { int x,y; }; // 아직 << 없음
struct Fancy {
    int v;
    std::string to_string() const { return "Fancy(" + std::to_string(v) + ")"; }
};

int main() {
    User u{42, "neo"};
    logln(u); // OK: Streamable

    Fancy f{7};
    logln(f); // OK: ToStringLike

    std::vector<int> a{1,2,3};
    log_range(a); std::cout << '\n'; // OK: int는 Streamable

    Point p{2,3};
    // log(p); // <- 컴파일 오류(의도대로 막힘)
}
```

# 6) 어떤 점을 배우는 예제인가?

* **커스텀 Concept**으로 “API 이용 권한”을 **선언적으로** 정의
* **오버로드 선택/삭제**로 정책을 분리 → **진단 메시지**가 좋아짐
* 필요시 **대체 정책**(to_string, range 로그)을 함께 제공
* `StreamInsertable<T, CharT>`처럼 **제네릭한 스트림 모델**을 도입해 확장성 확보

---

# 7) 자주 만나는 함정 & 팁

* `operator<<`는 **비멤버**로 정의(+ `friend`)하는 게 ADL에 유리합니다.
* 반드시 `<ostream>`/`<iostream>` 포함.
* 템플릿에서 반환형까지 제약(`-> std::same_as<...>`)을 붙이면 **의도와 다른 오버로드**를 조기에 차단합니다.
* `std::string`처럼 이미 Streamable인 타입이 Range이기도 합니다(문자 시퀀스). “문자열을 Range 로깅” 오버로드가 있으면 **모호성**이 생길 수 있으니, Range 로깅 컨셉에서 `!StringLike<R>`를 조건에 추가해 배제하세요.
