# 왜 `noexcept`를 제약에 넣나?

강한 예외 보장은 “실패(예외) 시 **관찰 가능한 상태가 전혀 바뀌지 않음**”을 뜻합니다.
템플릿 오버로드에서 **수정(in-place) 경로**를 허용하려면, 그 경로에 쓰이는 핵심 연산(할당/스왑/호출 등)이 **절대 던지지 않음**이 컴파일 타임에 보장돼야 합니다.
이를 `requires`-식의 **복합 요구**로 적습니다:

```cpp
// 표현이 잘 형성될 뿐 아니라, '던지지 않음'까지 검사
requires requires (T a, T b) {
    { std::ranges::swap(a, b) } noexcept;      // ← 여기!
    { a = std::move(b) } noexcept;             // ← 여기!
}
```

> 문법: `{ expr } noexcept -> 조건;`
> `noexcept`는 **표현식이 무조건 예외를 던지지 않음**을 요구합니다. (cppreference: Constraints & concepts / requires-expression)

# 실전 패턴 1: **nothrow move-assign** 경로 vs **nothrow swap** 경로

강한 보장을 지키는 대표 트릭은 두 가지입니다.

1. **이동 대입이 noexcept**면 그대로 **제자리 갱신**해도 강한 보장 유지 가능
2. 아니면 **스왑이 noexcept**라면 “copy/move-construct → `noexcept` 스왑”으로 **트랜잭션** 구현

둘 중 가능한 경로만 자동 선택하게 만듭니다.

```cpp
#include <concepts>
#include <utility>
#include <type_traits>
#include <ranges>
#include <string>

template <typename T>
concept NothrowMoveAssignable =
    std::is_nothrow_move_assignable_v<T>;

template <typename T>
concept NothrowSwappable =
    requires (T a, T b) {
        { std::ranges::swap(a, b) } noexcept;  // 복합 요구: 반드시 noexcept
    };

// 강한 예외 보장으로 x를 새 값으로 치환하는 API
// 경로 A: move-assign 자체가 noexcept면 제자리 갱신
template <typename T>
requires NothrowMoveAssignable<T>
void assign_strong(T& x, T new_value) noexcept {
    x = std::move(new_value);                  // 여기가 절대 안 던짐
}

// 경로 B: move-assign은 던질 수 있지만 swap이 noexcept면 copy-then-swap
template <typename T>
requires (!NothrowMoveAssignable<T> && NothrowSwappable<T>)
void assign_strong(T& x, const T& v) {
    T tmp = v;                                 // 여기서 예외 나면 x는 그대로(강한 보장)
    std::ranges::swap(x, tmp);                 // noexcept, 상태 전환 커밋
}

// 경로 C: 둘 다 아니면 사용 금지(진단 친절)
template <typename T>
requires (!NothrowMoveAssignable<T> && !NothrowSwappable<T>)
void assign_strong(T&, const T&) = delete;
```

### 셀프 테스트용 타입

```cpp
struct MoveNoexcept {
    int v{};
    MoveNoexcept& operator=(MoveNoexcept&&) noexcept { return *this; }
};
static_assert( NothrowMoveAssignable<MoveNoexcept> );

struct SwapNoexcept {
    int v{};
    SwapNoexcept& operator=(SwapNoexcept&&) { return *this; } // 던질 수 있음
    friend void swap(SwapNoexcept& a, SwapNoexcept& b) noexcept { std::swap(a.v, b.v); }
};
static_assert(!NothrowMoveAssignable<SwapNoexcept>);
static_assert( NothrowSwappable<SwapNoexcept> );

struct Bad {
    int v{};
    SwapNoexcept& operator=(SwapNoexcept&&) noexcept(false);  // 예: 던질 수 있고
    // swap도 제공 안 함 → 강한 보장 경로 없음
};
// assign_strong(Bad&, ...) 호출 시 컴파일 에러(의도한 진단)
```

# 실전 패턴 2: **콜렉션 갱신**의 트랜잭션 경로 선택

컨테이너를 “제자리로” 수정하는 경로는 **많은 연산(할당/스왑/연결/할당자/메모리)** 이 개입하므로, 강한 보장을 **조건부로** 제공하는 게 안전합니다.

```cpp
#include <vector>
#include <ranges>
#include <stdexcept>

template <typename T, std::ranges::input_range R>
requires std::same_as<std::remove_cvref_t<std::ranges::range_value_t<R>>, T>
void append_strong(std::vector<T>& dst, R&& src) {
    // 경로 1: 요소 이동이 noexcept이고 swap이 noexcept면,
    //         임시 없이 reserve 후 push_back으로 제자리 갱신 (부분 실패 시? → 안전하게 만들기 어려움)
    if constexpr (std::is_nothrow_move_constructible_v<T> &&
                  std::is_nothrow_move_assignable_v<T> &&
                  requires (std::vector<T>& a, std::vector<T>& b) { { a.swap(b) } noexcept; })
    {
        // 이 경로는 구현 난이도가 높아 예시에서는 생략하거나,
        // push_back이 던질 수 있음을 고려해 "임시 + swap"을 권장합니다.
    }
    else {
        // 경로 2(권장): 항상 강한 보장
        //  - 임시에 복사/이동 (실패하면 dst는 그대로)
        //  - 성공하면 dst와 임시를 noexcept 스왑
        std::vector<T> tmp = dst;                 // 1) 복사(예외 가능)
        for (auto&& e : src) tmp.push_back(e);    // 2) 임시 조작(예외 가능)
        dst.swap(tmp);                            // 3) 커밋(noexcept)
    }
}
```

> 컨테이너 갱신은 **임시 + 스왑**이 가장 간단한 강한 보장 패턴입니다.
> 제자리 경로를 여는 경우엔 `push_back/insert` 경로의 예외 가능성을 면밀히 따져야 합니다(할당자/재할당/요소 이동/복사 등). 실무에서는 대개 **임시+스왑**을 택합니다.

# 실전 패턴 3: **호출 정책**을 `noexcept`로 제약

콜백을 받는 API에서 “강한 보장을 유지하려면 콜백이 절대 던지지 않아야 함”을 캡처합니다.

```cpp
#include <concepts>
#include <utility>

template <typename F, typename... Args>
concept NothrowCallable =
    requires (F f, Args... args) {
        { std::invoke(f, args...) } noexcept;     // 복합 요구: 반드시 noexcept
    };

template <typename F, typename... Args>
requires NothrowCallable<F&, Args...>
void do_transaction(F&& op, Args&&... args) {
    // 트랜잭션 전 상태 스냅샷(또는 임시 객체 준비)
    // ...
    std::invoke(op, std::forward<Args>(args)...); // 절대 안 던짐 보장
    // 커밋
}
```

# 핵심 포인트 정리

* **복합 요구**의 `noexcept`로 “이 표현식은 절대 던지지 않음”을 제약 → **강한 보장 경로만 오버로드 후보**가 됨.
* 경로가 없으면 **삭제 오버로드**로 명확한 진단 제공.
* 컨테이너/복합 상태 갱신은 **임시 + `noexcept` 스왑**이 가장 쉬운 강한 보장 패턴.
* 콜백 기반 API는 콜백을 **`noexcept` 제약**으로 받으면 트랜잭션을 깔끔히 설계 가능.
