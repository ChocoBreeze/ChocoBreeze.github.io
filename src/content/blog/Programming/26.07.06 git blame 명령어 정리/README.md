`git blame`은 Git에서 **파일의 각 줄이 어떤 커밋에서, 누가, 언제 수정했는지 확인하는 명령어**입니다.

작업하다 보면 특정 코드가 왜 들어갔는지 궁금할 때가 있습니다.

예를 들어 다음과 같은 상황입니다.

```text
이 코드는 언제 추가된 걸까?
이 조건문은 왜 들어간 걸까?
이 줄을 수정한 커밋은 무엇일까?
이 변경이 어떤 작업과 관련된 것일까?
```

이때 사용할 수 있는 명령어가 `git blame`입니다.

기본 사용법은 다음과 같습니다.

```bash
git blame <file>
```

---

## 1. `git blame`이란?

`git blame`은 특정 파일의 각 줄마다 마지막으로 수정된 커밋 정보를 보여주는 명령어입니다.

예를 들어 `app.js` 파일이 있다고 해보겠습니다.

```bash
git blame app.js
```

그러면 각 줄 앞에 커밋 해시, 작성자, 날짜, 줄 번호 등이 표시됩니다.

예시:

```text
a1b2c3d4 (Hong 2026-07-01 10:15:22 +0900 1) function login() {
a1b2c3d4 (Hong 2026-07-01 10:15:22 +0900 2)   validateUser();
d4e5f6g7 (Kim  2026-07-02 14:30:11 +0900 3)   saveLoginHistory();
a1b2c3d4 (Hong 2026-07-01 10:15:22 +0900 4) }
```

이 결과를 보면 `saveLoginHistory();` 줄은 `d4e5f6g7` 커밋에서 `Kim`이 수정한 줄이라는 것을 알 수 있습니다.

즉, `git blame`은 파일의 줄 단위 변경 이력을 추적할 때 사용합니다.

---

## 2. `git blame <file>`

```bash
git blame <file>
```

`git blame <file>`은 지정한 파일의 각 줄이 마지막으로 수정된 커밋 정보를 보여줍니다.

예를 들어 `README.md` 파일의 줄별 변경 이력을 보고 싶다면 다음처럼 입력합니다.

```bash
git blame README.md
```

출력 예시는 다음과 같습니다.

```text
a1b2c3d4 (Hong 2026-07-01 09:10:11 +0900 1) # Project Title
a1b2c3d4 (Hong 2026-07-01 09:10:11 +0900 2)
d4e5f6g7 (Kim  2026-07-03 13:22:45 +0900 3) This project is a calendar app.
```

각 줄은 보통 다음 구조로 되어 있습니다.

```text
커밋해시 (작성자 날짜 시간 줄번호) 실제 파일 내용
```

예를 들어 다음 줄을 보면:

```text
d4e5f6g7 (Kim 2026-07-03 13:22:45 +0900 3) This project is a calendar app.
```

다음 의미로 이해할 수 있습니다.

```text
3번째 줄은 d4e5f6g7 커밋에서 마지막으로 수정되었다.
작성자는 Kim이다.
수정 시점은 2026-07-03 13:22:45이다.
```

`git blame`은 다음 상황에서 유용합니다.

```text
특정 코드가 언제 추가되었는지 확인할 때
특정 줄을 수정한 커밋을 찾고 싶을 때
코드 변경 이유를 추적하고 싶을 때
버그가 생긴 코드의 도입 시점을 확인하고 싶을 때
해당 줄과 관련된 커밋 메시지를 찾고 싶을 때
```

중요한 점은 `git blame`을 "누가 잘못했는지 찾는 명령어"로만 생각하면 안 된다는 것입니다.

이름은 blame이지만, 실제 목적은 책임 추궁이 아니라 **변경 맥락을 찾는 것**에 가깝습니다.

```text
누가 잘못했는지 찾기 X
왜 이 코드가 들어왔는지 추적하기 O
```

## 3. `git blame` 출력에서 커밋 확인하기

`git blame` 결과에서 커밋 해시를 확인한 뒤, 해당 커밋의 자세한 내용을 보고 싶다면 `git show`를 사용할 수 있습니다.

예를 들어 `git blame` 결과에서 다음 커밋을 발견했다고 해보겠습니다.

```text
d4e5f6g7 (Kim 2026-07-03 13:22:45 +0900 3) This project is a calendar app.
```

이 커밋의 자세한 내용을 보려면 다음 명령어를 실행합니다.

```bash
git show d4e5f6g7
```

그러면 해당 커밋의 메시지, 작성자, 날짜, 실제 변경 내용 등을 확인할 수 있습니다.

즉, 자주 쓰는 흐름은 다음과 같습니다.

```bash
git blame <file>
git show <commit-hash>
```

이렇게 보면 단순히 "누가 수정했는지"뿐만 아니라 "왜 수정했는지"도 커밋 메시지와 변경 내용을 통해 추적할 수 있습니다.

---

## 4. 줄 번호 기준으로 보기

파일 전체가 너무 길면 `git blame <file>`만 실행했을 때 결과가 너무 많이 나올 수 있습니다.

이럴 때는 특정 줄 범위만 확인하는 방식이 유용합니다.

```bash
git blame -L <start>,<end> <file>
```

예를 들어 `app.js` 파일의 10번째 줄부터 20번째 줄까지만 보고 싶다면 다음처럼 입력합니다.

```bash
git blame -L 10,20 app.js
```

이 명령어는 `app.js`의 10~20번째 줄에 대한 blame 결과만 보여줍니다.

```text
git blame -L 10,20 app.js
→ app.js 파일의 10번 줄부터 20번 줄까지만 추적
```

실제로는 전체 파일보다 문제가 되는 코드 주변만 보는 경우가 많기 때문에 `-L` 옵션을 함께 쓰면 훨씬 편합니다.

### 특정 함수 주변을 확인할 때

코드 리뷰나 버그 분석 중에는 특정 함수 안의 코드가 언제 바뀌었는지 보고 싶을 때가 많습니다.

예를 들어 `login()` 함수가 30~50번째 줄에 있다면 다음처럼 확인할 수 있습니다.

```bash
git blame -L 30,50 src/login.js
```

그 결과에서 의심되는 커밋 해시를 찾고:

```text
a1b2c3d4 (Hong 2026-07-01 09:10:11 +0900 35) if (user.isLocked()) {
```

해당 커밋을 자세히 확인합니다.

```bash
git show a1b2c3d4
```

이 흐름을 사용하면 코드 한 줄만 보는 것이 아니라, 해당 줄이 들어온 커밋 전체의 맥락을 확인할 수 있습니다.

## 5. `git blame`과 `git log` 차이

`git blame`과 `git log`는 모두 Git 기록을 확인할 때 사용하지만, 보는 관점이 다릅니다.

| 명령어                | 보는 기준    | 목적                   |
| ------------------ | -------- | -------------------- |
| `git log`          | 커밋 기준    | 커밋 기록 확인             |
| `git blame <file>` | 파일의 줄 기준 | 각 줄이 마지막으로 수정된 커밋 확인 |

예를 들어 `git log`는 다음과 같은 질문에 답하기 좋습니다.

```text
최근에 어떤 커밋들이 있었지?
이 브랜치에 어떤 작업이 들어왔지?
특정 커밋 메시지는 어디에 있지?
```

반면 `git blame`은 다음 질문에 답하기 좋습니다.

```text
이 파일의 25번째 줄은 어떤 커밋에서 수정됐지?
이 조건문은 언제 추가됐지?
이 코드와 관련된 커밋은 무엇이지?
```

즉, 커밋 흐름을 보고 싶으면 `git log`, 코드 줄의 출처를 보고 싶으면 `git blame`을 사용하면 됩니다.

---

## 6. `git blame`과 `git show`를 같이 쓰기

`git blame`은 줄별 커밋 해시를 찾는 데 좋고, `git show`는 그 커밋의 상세 내용을 보는 데 좋습니다.

그래서 두 명령어를 함께 사용하는 경우가 많습니다.

예를 들어 다음과 같이 사용합니다.

```bash
git blame -L 40,60 src/app.js
```

출력에서 의심되는 커밋 해시를 확인합니다.

```text
d4e5f6g7 (Kim 2026-07-03 13:22:45 +0900 45) return calculatePrice(order);
```

그다음 해당 커밋의 자세한 변경 내용을 확인합니다.

```bash
git show d4e5f6g7
```

이렇게 하면 다음 내용을 함께 볼 수 있습니다.

```text
어떤 줄이 문제인지
그 줄이 어떤 커밋에서 들어왔는지
그 커밋에서 다른 파일도 함께 수정했는지
커밋 메시지는 무엇인지
```

단순히 해당 줄만 보는 것보다 훨씬 정확하게 변경 이유를 파악할 수 있습니다.

---

## 7. 파일 이름이 바뀐 경우 주의

`git blame`은 기본적으로 현재 파일을 기준으로 줄별 변경 이력을 보여줍니다.

그런데 파일 이름이 변경되었거나 이동된 적이 있다면, 과거 이력이 끊겨 보이는 것처럼 느껴질 수 있습니다.

이럴 때는 파일 이름 변경을 따라가며 볼 수 있습니다.

```bash
git blame --follow <file>
```

예를 들어 `old-login.js`가 `login.js`로 이름이 바뀐 이력이 있다면 다음처럼 사용할 수 있습니다.

```bash
git blame --follow src/login.js
```

다만 파일 이동이나 이름 변경이 복잡한 경우에는 `git log --follow <file>`로 먼저 파일 이력을 확인하는 것도 좋습니다.

```bash
git log --follow --oneline src/login.js
```

---

## 8. 공백 변경 때문에 보기 어려울 때

가끔 코드를 실제로 고친 것이 아니라, 들여쓰기나 줄 정리 때문에 blame 결과가 크게 바뀌어 보일 수 있습니다.

예를 들어 코드 포맷터를 적용하면 많은 줄의 마지막 수정 커밋이 formatter 커밋으로 표시될 수 있습니다.

이럴 때는 공백 변경을 무시하고 blame을 볼 수 있습니다.

```bash
git blame -w <file>
```

예를 들어:

```bash
git blame -w src/app.js
```

`-w`는 whitespace 변경을 무시한다는 의미입니다.

즉, 실제 로직 변경을 추적하고 싶은데 공백 변경 때문에 결과가 헷갈릴 때 사용할 수 있습니다.

---

## 9. `git blame`을 사용할 때의 태도

`git blame`이라는 이름 때문에 "누가 이 코드를 잘못 짰는지 찾는 명령어"처럼 느껴질 수 있습니다.

하지만 실제로는 협업에서 문제 원인을 이해하기 위한 도구로 사용하는 것이 좋습니다.

예를 들어 어떤 코드가 의심될 때 바로 사람을 지목하기보다, 다음 순서로 보는 것이 좋습니다.

```text
1. git blame으로 해당 줄의 커밋 해시를 찾는다.
2. git show로 커밋 전체 내용을 확인한다.
3. 커밋 메시지와 주변 변경 사항을 본다.
4. 필요하면 관련 이슈, PR, 리뷰 내용을 함께 확인한다.
```

코드는 당시 요구사항이나 상황에 맞춰 작성되었을 수 있습니다.

그래서 `git blame`은 사람을 탓하기 위한 도구가 아니라, **변경의 이유와 맥락을 찾는 도구**로 사용하는 것이 좋습니다.

## 10. 자주 쓰는 흐름

파일 전체의 줄별 변경 이력을 확인할 때:

```bash
git blame <file>
```

특정 줄 범위만 확인할 때:

```bash
git blame -L <start>,<end> <file>
```

예시:

```bash
git blame -L 10,30 src/app.js
```

blame으로 찾은 커밋을 자세히 볼 때:

```bash
git show <commit-hash>
```

파일 이름 변경 이력을 따라가며 보고 싶을 때:

```bash
git blame --follow <file>
```

공백 변경을 무시하고 보고 싶을 때:

```bash
git blame -w <file>
```

## 11. 명령어 정리

| 명령어                                 | 의미                          |
| ----------------------------------- | --------------------------- |
| `git blame <file>`                  | 파일의 각 줄이 마지막으로 수정된 커밋 정보 확인 |
| `git blame -L <start>,<end> <file>` | 특정 줄 범위만 blame 확인           |
| `git blame --follow <file>`         | 파일 이름 변경 이력을 따라가며 blame 확인  |
| `git blame -w <file>`               | 공백 변경을 무시하고 blame 확인        |
| `git show <commit-hash>`            | blame으로 찾은 커밋의 상세 내용 확인     |

```text
git blame <file>
→ 파일의 각 줄이 어떤 커밋에서 수정됐는지 확인한다.

git blame -L 10,20 <file>
→ 특정 줄 범위만 확인한다.

git blame --follow <file>
→ 파일 이름 변경 이력을 따라가며 확인한다.

git blame -w <file>
→ 공백 변경을 무시하고 확인한다.

git show <commit-hash>
→ 해당 커밋의 자세한 내용을 확인한다.
```

## 12. 자주 헷갈리는 상황

### `git blame`은 누가 잘못했는지 찾는 명령어인가?

이름은 blame이지만, 실제로는 책임을 묻기 위한 명령어라기보다 변경 이력을 추적하는 명령어입니다.

특정 줄이 언제, 어떤 커밋에서 수정되었는지 확인하고, 그 변경의 맥락을 찾는 데 사용합니다.

```text
사람을 탓하기 위한 명령어 X
변경 이력을 추적하기 위한 명령어 O
```

### `git blame` 결과의 작성자가 현재 코드 작성자와 항상 같을까?

항상 그렇지는 않습니다.

`git blame`은 해당 줄을 마지막으로 수정한 커밋 정보를 보여줍니다.

처음 작성한 사람이 아니라, 마지막으로 그 줄을 건드린 사람이 표시될 수 있습니다.

예를 들어 원래 작성자는 A였지만, B가 공백이나 작은 문구를 수정했다면 blame에는 B가 표시될 수 있습니다.

이럴 때는 `git show <commit-hash>`로 실제 변경 내용을 확인하는 것이 좋습니다.

### `git blame`만 보면 변경 이유까지 알 수 있을까?

아닙니다.

`git blame`은 해당 줄과 관련된 커밋 정보를 알려줍니다.

하지만 왜 변경했는지는 커밋 메시지, 변경된 다른 파일, 이슈나 PR 내용을 함께 봐야 더 정확히 알 수 있습니다.

그래서 보통은 다음 흐름으로 확인합니다.

```bash
git blame <file>
git show <commit-hash>
```

### 파일 이름이 바뀌면 과거 이력이 안 보일 수 있나?

그럴 수 있습니다.

파일 이름이 변경되었거나 이동된 경우에는 다음 옵션을 함께 사용할 수 있습니다.

```bash
git blame --follow <file>
```

파일 자체의 이동 이력을 먼저 보고 싶다면 다음 명령어도 도움이 됩니다.

```bash
git log --follow --oneline <file>
```

## 마무리 정리

`git blame <file>`은 특정 파일의 각 줄이 어떤 커밋에서 마지막으로 수정되었는지 확인하는 명령어입니다.

파일의 줄 단위로 커밋 해시, 작성자, 날짜, 줄 번호를 확인할 수 있어서 특정 코드가 언제 들어왔는지 추적할 때 유용합니다.

다만 `git blame`은 사람을 탓하기 위한 명령어라기보다, 변경의 이유와 맥락을 찾기 위한 도구로 사용하는 것이 좋습니다.

blame 결과에서 커밋 해시를 찾은 뒤에는 `git show <commit-hash>`로 해당 커밋의 상세 내용을 함께 확인하면 더 정확하게 이해할 수 있습니다.

한 문장으로 정리하면 다음과 같습니다.

> `git blame`은 파일의 각 줄이 어떤 커밋에서 마지막으로 수정되었는지 확인해, 코드 변경의 출처와 맥락을 추적할 때 사용하는 명령어이다.
