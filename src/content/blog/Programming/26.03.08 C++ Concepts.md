# Concepts란 무엇인가?

**Concepts**는 *템플릿 인자*가 만족해야 하는 **조건(제약)** 을 이름 붙여 표현하는 언어 기능입니다.
목표는 세 가지입니다.

1. **의도 명확화**: 템플릿이 기대하는 타입/표현을 코드로 명시
2. **오류 진단 향상**: SFINAE 트릭보다 훨씬 읽기 쉬운 에러 메시지
3. **오버로드 선택 개선**: 더 구체적인 제약(=만족도가 높은 제약)이 우선됩니다(“subsumption”)

---

# Concepts를 쓰는 방법 요약

## 1) `concept`으로 이름 붙이기

```cpp
#include <type_traits>

template <typename T>
concept Integral = std::is_integral_v<T>;  // 불 표현식이면 OK

static_assert(Integral<int>);
static_assert(!Integral<double>);
```

## 2) `requires` 절(클로저)로 제약 달기

```cpp
template <typename T>
requires Integral<T>                      // 템플릿에 조건 부여
T add(T a, T b) { return a + b; }
```

같은 의미를 “후행 requires”로도 쓸 수 있습니다.

```cpp
template <typename T>
T add2(T a, T b) requires Integral<T> {   // 시그니처 뒤에 배치
    return a + b;
}
```

## 3) 축약 템플릿(Abbreviated templates)

**constrained auto**를 쓰면 함수 템플릿을 간결하게:

```cpp
Integral auto add3(Integral auto a, Integral auto b) { return a + b; }
// 위와 동일하게 T가 integral인 경우에만 인스턴스화
```

## 4) `requires`-expression (요구식) — 표현 가능성 검사

멤버/연산자/반환 타입 제약을 **표현식 블록**으로 자세히 기술합니다.

요구식의 요소:

* **타입 요구(type requirement)**: `typename T::value_type;`
* **복합 요구(compound requirement)**: `{ expr } noexcept -> concept/trait;`
* **중첩 요구(nested requirement)**: `requires (bool-expression);`

```cpp
#include <concepts>    // std::same_as 등

template <typename T>
concept HasPushBack = requires(T x, typename T::value_type v) {
    typename T::value_type;                             // 타입 요구
    { x.push_back(v) } -> std::same_as<void>;           // 복합 요구(반환 타입)
    requires (!std::is_const_v<T>);                     // 중첩 요구(임의의 불식)
};
```

# 표준 라이브러리의 대표 Concept

* `<concepts>`: `std::integral`, `std::floating_point`, `std::same_as`, `std::convertible_to`, `std::invocable` …
* `<iterator>`/`<ranges>`: `std::input_iterator`, `std::random_access_iterator`, `std::ranges::range`, `std::ranges::random_access_range` …
* `<ranges>` 알고리즘과 함께 강력한 제약 기반 오버로드를 제공합니다.

```cpp
#include <concepts>
#include <ranges>
#include <vector>
#include <list>

// random_access_range에만 허용
template <std::ranges::random_access_range R>
void fast_only(R&& r) { /* ... */ }

// 모든 range에 허용
template <std::ranges::range R>
void generic(R&& r) { /* ... */ }
```

# 실전 예제 1: 안전한 수치 연산 API

```cpp
#include <concepts>
#include <limits>
#include <stdexcept>

template <std::integral T>
T saturated_add(T a, T b) {
    if ((b > 0) && (a > std::numeric_limits<T>::max() - b)) {
        return std::numeric_limits<T>::max();
    }
    if ((b < 0) && (a < std::numeric_limits<T>::min() - b)) {
        return std::numeric_limits<T>::min();
    }
    return a + b;
}

// 부동소수점 버전
template <std::floating_point T>
T saturated_add(T a, T b) {
    T r = a + b;
    if (!std::isfinite(r)) throw std::overflow_error{"float overflow"};
    return r;
}
```

* 정수/실수에 대해 **오버로드를 Concepts로 분리**하므로 의도가 명확하고 모호성이 없습니다.
* 더 구체적인 제약이 있으면 그 오버로드가 우선됩니다(부분 순서/서브섬션).

# 실전 예제 2: `requires`-expression으로 멤버 요구를 기술

```cpp
#include <concepts>
#include <utility>

template <typename T>
concept Indexable = requires(T t, std::size_t i) {
    { t[i] } -> std::same_as<typename T::reference>;        // []의 반환 타입 제약
    { std::as_const(t)[i] } -> std::same_as<typename T::const_reference>;
    requires requires (T u) { u.size(); };                  // 중첩 요구: size()가 존재
};

template <Indexable C>
auto front(C& c) -> typename C::reference {
    if (c.size() == 0) throw std::out_of_range{"empty"};
    return c[0];
}
```

# 실전 예제 3(심화): 제약 기반 알고리즘 오버로드 + 서브섬션

**무작위 접근 가능(range)** 이면 더 빠른 경로, 아니면 일반 경로:

```cpp
#include <algorithm>
#include <ranges>
#include <vector>
#include <list>

// 더 구체적인 제약(=무작위 접근) 오버로드
template <std::ranges::random_access_range R>
requires std::sortable<std::ranges::iterator_t<R>>
void my_sort(R&& r) {
    std::sort(std::ranges::begin(r), std::ranges::end(r));  // O(n log n)
}

// 더 포괄적인 제약(모든 range)
template <std::ranges::range R>
requires std::sortable<std::ranges::iterator_t<R>>
void my_sort(R&& r) {
    // 임의 접근이 아니라면 범용 but 느린 알고리즘(예: insertion sort 데모)
    for (auto it = std::ranges::begin(r); it != std::ranges::end(r); ++it) {
        std::ranges::iter_value_t<R> key = *it;
        auto jt = it;
        while (jt != std::ranges::begin(r) && *std::prev(jt) > key) {
            *jt = *std::prev(jt);
            --jt;
        }
        *jt = key;
    }
}
```

* **서브섬션 규칙** 때문에 `random_access_range` 버전이 가능하면 그쪽이 자동으로 선택됩니다.
* 이전 세대의 SFINAE/tag-dispatch 없이도 **읽기 쉽고 강력한 오버로드** 구성이 됩니다.

# 실전 예제 4(심화): 행렬 덧셈 – “형태/연산 가능성”을 동시에 제약

```cpp
#include <concepts>
#include <cstddef>
#include <vector>
#include <type_traits>

template <typename T, std::size_t R, std::size_t C>
struct Matrix {
    using value_type = T;
    std::vector<T> data;
    static constexpr std::size_t rows = R, cols = C;

    T& operator()(std::size_t r, std::size_t c) { return data[r*C + c]; }
    const T& operator()(std::size_t r, std::size_t c) const { return data[r*C + c]; }
};

template <typename M>
concept MatrixLike = requires (M m) {
    typename M::value_type;
    { M::rows } -> std::convertible_to<std::size_t>;
    { M::cols } -> std::convertible_to<std::size_t>;
    { m(0,0) } -> std::same_as<typename M::value_type&> || std::same_as<const typename M::value_type&>;
};

template <MatrixLike M>
using scalar_t = typename M::value_type;

template <MatrixLike A, MatrixLike B>
requires (A::rows == B::rows) && (A::cols == B::cols) &&
         requires (scalar_t<A> a, scalar_t<B> b) { a + b; }   // 원소 덧셈 가능
auto add(const A& a, const B& b) {
    using R = std::common_type_t<scalar_t<A>, scalar_t<B>>;
    Matrix<R, A::rows, A::cols> r{{}};
    r.data.resize(A::rows * A::cols);
    for (std::size_t i = 0; i < A::rows; ++i)
        for (std::size_t j = 0; j < A::cols; ++j)
            r(i,j) = static_cast<R>(a(i,j)) + static_cast<R>(b(i,j));
    return r;
}
```

설명 포인트:

* `MatrixLike`로 **형태적 요구(멤버/호출 가능성)** 를 기술.
* `add`에서 **치수 동일성**(정수 NTTP 비교)과 **원소 연산 가능성**을 동시 제약.
* 실패 시 “`add`의 요구 조건을 만족하지 않음” 같은 **명확한 컴파일 오류**를 얻습니다.

# 실무 팁 & 주의할 점

* **과제약(over-constraining)** 을 피하세요. 가능한 한 *의미 있는 최소 제약*을 표기합니다.
* **개념은 불(boolean) 값**입니다. 필요한 경우 조합(`&&`, `||`, `!`)해 쓰세요.
* **진단 품질**: SFINAE보다 훨씬 읽을 만한 에러가 나오므로, 새 코드라면 SFINAE보다 Concepts를 우선 고려하세요.
* **가독성**: 복잡한 `requires`-expression은 **작게 쪼갠 named concept**으로 나눠 재사용하면 훨씬 깨끗해집니다.
* **성능**: Concepts 자체는 런타임 오버헤드가 없습니다. 다만 오버로드 구조가 달라지면 선택되는 구현이 달라질 수 있습니다.
