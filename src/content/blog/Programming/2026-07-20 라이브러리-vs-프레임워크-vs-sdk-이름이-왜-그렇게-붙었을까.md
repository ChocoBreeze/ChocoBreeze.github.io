> MCP 강의를 듣다가 "MCP Python SDK"라는 이름에서 시작된 질문. `@mcp.tool()`의 `@`가 SDK인가? LangChain도 SDK인가? — 꼬리를 물다 보니 세 용어의 구분 기준이 각각 다르다는 걸 알게 되어 정리한다.

---

## 세 용어, 세 가지 다른 기준

| 개념 | 구분 기준 | 예시 |
|---|---|---|
| **라이브러리** | 코드 묶음. **내 코드가 호출한다** | requests, pandas, Jackson |
| **프레임워크** | 구조를 제공. **프레임워크가 내 코드를 호출한다** (제어의 역전) | Spring, LangChain, LangGraph |
| **SDK** | 특정 **플랫폼·서비스·프로토콜을 대상으로** 개발하기 위한 키트 | JDK, AWS SDK, MCP Python SDK |

핵심은 세 용어가 **서로 다른 축의 질문**에 답한다는 것이다.

- 라이브러리 vs 프레임워크 → "**누가 누구를 호출하는가?**"
- SDK인가 아닌가 → "**명확한 대상이 있는가?**"

---

## 1. 라이브러리 vs 프레임워크: 호출 방향

**라이브러리**는 내가 필요할 때 갖다 쓰는 코드다. `requests.get(url)` — 호출의 주도권이 내 코드에 있다.

**프레임워크**는 반대다. 전체 구조와 실행 흐름을 프레임워크가 쥐고 있고, 내 코드는 그 틀의 빈칸에 끼워진다. Spring이 대표적이다:

```java
@GetMapping("/users/{id}")
public User getUser(@PathVariable Long id) { ... }
```

이 메서드를 내가 직접 호출하는 일은 없다. HTTP 요청이 오면 **Spring이 이 메서드를 호출한다**. 이 뒤집힌 관계가 **제어의 역전(IoC, Inversion of Control)**이고, 프레임워크의 정체성이다. "Don't call us, we'll call you"라는 할리우드 원칙으로도 불린다.

LangChain/LangGraph가 프레임워크인 이유도 같다. LangGraph에서 나는 노드 함수를 정의할 뿐, 그 함수들을 언제 어떤 순서로 실행할지는 그래프 런타임이 결정한다.

### "코드가 나를 호출한다"는 느낌 — main()의 소유권

호출 방향이 뒤집힌다는 게 처음엔 감이 안 올 수 있는데, **`main()`이 누구 것인가**로 생각하면 명확해진다.

라이브러리 세계에서는 `main()`이 내 것이다. 프로그램의 시작부터 끝까지 실행 순서를 내가 지휘하고, 라이브러리는 필요할 때 불러 쓰는 부하다. 프레임워크 세계에서는 `main()`과 실행 루프를 **프레임워크가 소유**한다. 나는 "이런 상황이 오면 이 함수를 실행해줘"라고 **등록만** 하고, 언제·몇 번·어떤 순서로 호출될지는 프레임워크가 결정한다.

사실 이 구조는 C++에서도 이미 겪는다 — 이름을 안 붙였을 뿐이다. `qsort`의 비교 함수가 최소 단위의 예시다:

```cpp
int compare(const void* a, const void* b) {
    return (*(int*)a - *(int*)b);
}

qsort(arr, n, sizeof(int), compare);  // compare를 "건네주기만" 한다
```

`compare`를 내가 호출한 적은 없다. **qsort가 정렬 도중 자기가 필요할 때마다 알아서 호출**하고, 몇 번 호출될지도 나는 모른다. 이런 함수를 **콜백(callback)**이라 부르며, Win32의 `WndProc`이나 Qt의 시그널/슬롯도 같은 구조다. **프레임워크란 이 콜백 구조를 애플리케이션 전체 규모로 확장한 것**이다 — 함수 하나가 아니라, 앱을 구성하는 내 코드 전부가 등록된 콜백이 된다.

그리고 등록할 때는 보통 **"어떤 조건/이벤트가 오면"과 짝지어서** 등록한다. 어노테이션/데코레이터 문법의 정체가 바로 이 "조건 → 내 함수" 매핑을 프레임워크의 라우팅 테이블에 써넣는 것이다.

| 등록 문법 | 짝지어진 조건 |
|---|---|
| `@GetMapping("/users")` | "이 URL로 GET 요청이 오면" |
| `@mcp.tool(name="read_doc_contents")` | "이 이름의 CallToolRequest가 오면" |
| `qsort(..., compare)` | "정렬 중 두 원소를 비교해야 하면" |

프레임워크가 이렇게까지 하는 이유는 단순하다. 서버의 골격(요청 대기 루프, 파싱, 에러 응답, 동시성)은 어떤 서버든 똑같고, 매번 달라지는 건 "요청이 왔을 때 뭘 할 것인가"뿐이다. 반복되는 골격을 프레임워크가 통째로 가져가고 나는 달라지는 부분만 빈칸 채우듯 등록하는 것 — **"코드 재사용"을 넘어 "제어 흐름 재사용"까지 간 것**이 프레임워크의 본질이다.

## 2. SDK: 대상이 있는 연장통

**SDK(Software Development Kit)**는 호출 방향이 아니라 "무엇을 대상으로 하는가"로 정의된다. 특정 플랫폼(iOS SDK), 특정 서비스(AWS SDK), 특정 프로토콜(MCP SDK)처럼 **명확한 대상**이 있고, 그 대상을 상대로 개발하는 데 필요한 것들을 세트로 묶은 것이다.

그리고 SDK는 라이브러리보다 **큰** 개념이다.

```
SDK = 라이브러리 + 개발 지원 도구 일체 (문서, 예제, CLI, 디버거...)
```

Java 개발자라면 사실 매일 SDK를 쓰고 있었다 — **JDK(Java Development Kit)**가 정확히 이 구조다. 표준 라이브러리(코드) + `javac`, `jar`, `jdb`(도구)가 합쳐져 "Kit"이라는 이름이 붙었다.

MCP Python SDK도 마찬가지다:

- `FastMCP`, `ClientSession` 클래스 → 라이브러리 부분
- `mcp dev server.py` 명령어(CLI), Inspector 연동 → 코드를 넘어서는 도구 부분

import해서 쓰는 코드만 있었다면 "mcp 라이브러리"라고 불렸을 것이다. CLI 같은 도구까지 세트라서 SDK다.

### 여담: 데코레이터는 SDK가 아니다

```python
@mcp.tool(name="read_doc_contents", ...)
def read_document(...): ...
```

이 코드에서 `@`는 Python 언어 자체의 **데코레이터 문법**이고, SDK는 그 뒤의 `mcp.tool`(= `FastMCP` 인스턴스의 메서드)이다. 관계를 정확히 쓰면: 데코레이터 문법은 **내 함수를 SDK에게 건네주는 입구**이고, 함수를 받아 스키마를 생성하고 tool로 등록하는 실제 능력이 SDK다. Spring의 `@` 어노테이션 문법(Java 기능)과 `GetMapping`(Spring 제공)의 관계와 정확히 같다.

## 3. 검증: LangChain 생태계의 명명법

이 구분이 실제로 지켜지는 좋은 사례가 LangChain 생태계다. 같은 회사 제품인데 이름이 갈린다.

| 패키지 | 역할 | 분류 |
|---|---|---|
| `langchain`, `langgraph` | LLM 앱의 구조·실행 흐름 제공 (내 코드를 런타임이 호출) | **프레임워크** |
| `langgraph-sdk` | LangGraph **Platform**(배포 서비스)에 접속하는 클라이언트 | **SDK** |
| `langsmith` | LangSmith(트레이싱 **서비스**)에 데이터를 보내는 클라이언트 | **SDK** |

"앱의 구조를 잡아주는 것"은 프레임워크, "특정 서비스에 접속하게 해주는 것"은 SDK — 명명 원칙이 일관되게 적용되어 있다. LangChain이 특정 LLM에 종속되지 않는 것(OpenAI든 Anthropic이든 교체 가능)도 "명확한 대상"이 없다는 뜻이므로 SDK가 아닌 이유와 통한다.

## 정리

두 줄로 압축하면 이렇다.

> - **라이브러리**: 내가 원하는 시점에 **내가 호출**한다.
> - **프레임워크**: 내가 원하는 동작을 함수로 **미리 등록**한다. 실제 호출 시점은 프레임워크가 결정한다.

판단 기준까지 포함하면:

> - **누가 호출하나?** — 내가 호출하면 라이브러리, 걔가 나를 호출하면 프레임워크
> - **명확한 대상(플랫폼·서비스·프로토콜)이 있나?** — 있으면 SDK, 없으면 아님
> - **SDK ⊃ 라이브러리** — 코드에 문서·CLI·디버거 같은 연장까지 묶이면 Kit

느낌으로는 이렇게 기억해도 좋다: 라이브러리를 쓸 때 나는 **기계를 조종하는 사람**이고, 프레임워크를 쓸 때 나는 **기계에 끼워지는 부품**이다.

재미있는 건, 이 개념을 몰라도 이미 프레임워크식으로 도구를 만들고 있었을 가능성이 높다는 것이다. 예컨대 Claude Code의 Hook(PreToolUse로 특정 tool 실행 차단 같은 것)이 정확히 이 구조다 — "커밋 tool이 실행되려는 순간(조건)이 오면 내 스크립트를 실행해줘(등록된 동작)"이고, 호출 시점은 Claude Code가 결정한다. Spring MVC의 `@GetMapping` 메서드들도 전부, 내가 호출한 적 없는 등록된 콜백들이다.

물론 현실에서는 마케팅적 혼용이 많다. 문서에서는 framework라 부르는데 패키지명은 sdk인 경우도 있다. 칼같은 분류보다는 "이 이름이 왜 붙었을까"를 읽어내는 감각으로 쓰는 게 실용적이다 — MCP **Python SDK**라는 이름 하나에서 "아, MCP라는 프로토콜이 대상이고, 코드 외에 도구도 같이 주는구나"까지 읽어낼 수 있으면 충분하다.
