## `co_await`에서 `coroutine_handle`까지

앞선 글에서는 비동기 실행의 공통 구조를 살펴봤다.

핵심은 다음과 같았다.

```text
현재 실행 상태를 저장한다.
→ 결과를 기다리는 동안 실행권을 돌려준다.
→ 결과가 준비되면 저장된 위치에서 다시 실행한다.
```

이 구조를 구현하려면 최소한 다음 요소가 필요하다.

```text
함수의 실행 상태를 보관할 공간
어디까지 실행했는지를 나타내는 상태
코루틴을 중단하는 방법
중단된 코루틴을 재개하는 방법
결과와 예외를 전달하는 방법
코루틴의 수명을 관리하는 방법
```

C++20은 이 문제를 해결하기 위해 코루틴 기능을 언어에 추가했다.

다만 여기서 먼저 분명히 해야 할 점이 있다.

> C++20 코루틴은 완성된 비동기 프레임워크가 아니다.

C++20은 함수가 실행 도중 중단되고 다시 재개될 수 있도록 만드는 **언어 차원의 변환 규약**을 제공한다.

하지만 다음 기능은 표준이 직접 제공하지 않는다.

```text
이벤트 루프
네트워크 비동기 I/O
스레드 풀
Task 스케줄러
취소 처리
타임아웃
표준 task<T> 타입
```

즉, C++20은 코루틴의 엔진 부품을 제공하지만, 완성된 자동차까지 제공하지는 않는다.

이 글에서는 다음과 같은 간단한 코루틴을 기준으로 내부 구조를 살펴본다.

```cpp
task<int> calculate() {
    int value = co_await async_operation();
    co_return value * 2;
}
```

겉보기에는 짧은 코드지만, 내부에서는 일반 함수보다 훨씬 많은 일이 일어난다.

---

## 1. 어떤 함수가 코루틴이 되는가

C++ 함수 안에 다음 키워드 중 하나가 사용되면 컴파일러는 해당 함수를 코루틴으로 변환한다.

```cpp
co_await
co_yield
co_return
```

예를 들어 다음 함수는 코루틴이다.

```cpp
task<int> calculate() {
    int value = co_await async_operation();
    co_return value * 2;
}
```

반면 다음 함수는 반환 타입이 `task<int>`라고 해도 코루틴이라고 단정할 수 없다.

```cpp
task<int> calculate() {
    return make_completed_task(10);
}
```

코루틴 여부를 결정하는 것은 반환 타입 이름이 아니라, 함수 본문에서 코루틴 키워드를 사용하는지 여부다.

컴파일러는 코루틴 함수를 일반 함수처럼 그대로 컴파일하지 않는다.

대신 개념적으로 다음 요소를 생성한다.

```text
코루틴 프레임을 생성하는 코드
코루틴을 처음 실행하는 코드
중단 상태를 저장하는 코드
재개 시 실행할 코드
코루틴을 파괴하는 코드
결과와 예외를 관리하는 코드
```

즉, 코루틴 함수는 컴파일 과정에서 상태 머신과 수명 관리 코드로 확장된다.

---

## 2. 코루틴 함수는 일반 함수와 무엇이 다른가

일반 함수가 호출되면 함수 실행에 필요한 지역변수와 반환 주소 등이 호출 스택에 저장된다.

```cpp
int add(int a, int b) {
    int result = a + b;
    return result;
}
```

실행 구조는 단순하다.

```text
함수 호출
→ 스택 프레임 생성
→ 함수 실행
→ 결과 반환
→ 스택 프레임 제거
```

그런데 코루틴은 함수 실행 도중 호출자에게 제어권을 돌려준 뒤, 나중에 다시 실행될 수 있어야 한다.

```cpp
task<int> calculate() {
    int value = co_await async_operation();
    co_return value * 2;
}
```

`co_await`에서 함수가 중단된다면 `value`를 저장할 위치와 재개할 코드 위치가 필요하다.

함수가 중단되었다고 해서 실행 상태를 제거하면 안 된다.

그래서 컴파일러는 코루틴의 상태를 저장하는 별도의 객체를 만든다.

이를 **코루틴 프레임**이라고 한다.

---

## 3. Coroutine Frame

코루틴 프레임은 코루틴이 중단된 뒤에도 유지되어야 하는 실행 상태를 보관한다.

개념적으로는 다음과 같은 구조를 생각할 수 있다.

```cpp
struct CalculateFrame {
    int state;
    int value;

    PromiseType promise;
    Awaiter operation_awaiter;

    std::exception_ptr exception;
};
```

실제 프레임의 형태는 컴파일러와 최적화 결과에 따라 다르다.

하지만 보통 다음 정보가 필요하다.

```text
함수의 매개변수
중단 시점에도 살아 있어야 하는 지역변수
현재 실행 위치
promise 객체
기다리고 있는 awaiter
예외 상태
재개 함수와 파괴 함수에 필요한 정보
```

일반 함수의 지역변수는 보통 스택에 저장되지만, 코루틴의 지역변수 중 중단을 넘어 살아 있어야 하는 값은 코루틴 프레임에 저장될 수 있다.

다음 코드를 보자.

```cpp
task<int> calculate() {
    int base = 10;
    int value = co_await async_operation();
    co_return base + value;
}
```

`base`는 `co_await` 이후에도 필요하다.

따라서 코루틴이 중단되는 동안에도 보존되어야 한다.

```text
실행 시작
→ base = 10
→ co_await에서 중단
→ base를 코루틴 프레임에 보존
→ 재개
→ base + value 계산
```

반대로 어떤 지역변수가 첫 번째 중단 지점 이전에만 사용되고 이후에는 필요 없다면, 컴파일러는 반드시 프레임에 저장하지 않을 수도 있다.

중요한 것은 코루틴 프레임이 소스 코드의 모든 지역변수를 기계적으로 복사한 구조체라는 뜻은 아니라는 점이다.

> 코루틴 프레임은 코루틴을 나중에 재개하는 데 필요한 상태를 저장하는 실행 객체다.

---

## 4. 코루틴 프레임은 어디에 생성되는가

코루틴 프레임은 일반적으로 동적 저장 영역에 할당될 수 있다.

즉, 개념적으로는 힙 할당과 비슷하게 생각할 수 있다.

```cpp
auto* frame = new CalculateFrame{};
```

하지만 항상 실제 힙 할당이 발생한다고 단정하면 안 된다.

컴파일러는 코루틴의 수명과 호출 관계를 분석해 할당을 제거하거나 호출자의 저장 공간에 포함할 수 있다.

즉, 정신 모델은 다음과 같다.

```text
기본적으로 함수 호출 스택보다 긴 수명이 필요한 별도 프레임이 존재한다.
```

실제 물리적인 메모리 배치는 최적화 결과에 따라 달라질 수 있다.

이 점은 C++ 객체 수명 관점에서 중요하다.

코루틴이 완료되었거나 더 이상 필요하지 않을 때는 코루틴 프레임을 반드시 파괴해야 한다.

그렇지 않으면 프레임 내부의 지역 객체와 메모리가 계속 남을 수 있다.

## 5. Coroutine State Machine

코루틴은 실행 위치를 저장하기 위해 상태 머신처럼 변환된다.

다음 코드를 보자.

```cpp
task<int> calculate() {
    int base = 10;
    int value = co_await async_operation();
    co_return base + value;
}
```

개념적으로는 다음과 같은 재개 함수가 만들어질 수 있다.

```cpp
void resume_calculate(CalculateFrame& frame) {
    switch (frame.state) {
    case 0:
        frame.base = 10;
        frame.operation_awaiter = get_awaiter(async_operation());

        if (!frame.operation_awaiter.await_ready()) {
            frame.state = 1;
            frame.operation_awaiter.await_suspend(
                coroutine_handle_for(frame)
            );
            return;
        }

        [[fallthrough]];

    case 1:
        frame.value =
            frame.operation_awaiter.await_resume();

        frame.promise.return_value(
            frame.base + frame.value
        );

        frame.state = 2;
        return;
    }
}
```

실제 생성 코드는 이보다 복잡하지만, 핵심 구조는 같다.

```text
state = 0
코루틴 시작

state = 1
첫 번째 co_await 이후 재개

state = 2
코루틴 완료
```

코루틴을 재개한다는 것은 개념적으로 다음과 같다.

```cpp
switch (frame.state) {
    // 저장된 상태에 따라 실행 위치 선택
}
```

그래서 코루틴은 "스택을 그대로 얼렸다가 다시 녹이는 함수"라기보다 다음 구조에 더 가깝다.

> 살아 있어야 하는 데이터를 프레임에 저장하고, 상태 값에 따라 다음 코드 조각을 실행하는 상태 머신

## 6. 반환 타입과 `promise_type`

C++ 코루틴에서 가장 처음 혼란스러운 부분은 `promise_type`이다.

이름 때문에 JavaScript의 `Promise`와 같은 비동기 결과 객체로 오해하기 쉽다.

하지만 C++ 코루틴의 `promise_type`은 조금 다른 역할을 한다.

> `promise_type`은 코루틴과 외부 반환 객체 사이의 동작을 정의하는 제어 객체다.

예를 들어 반환 타입이 다음과 같다고 하자.

```cpp
template <typename T>
class task {
public:
    struct promise_type;
};
```

코루틴 함수의 반환 타입이 `task<int>`이면, 컴파일러는 대략 다음 타입을 찾는다.

```cpp
task<int>::promise_type
```

이 `promise_type`은 다음과 같은 정책을 제공한다.

```text
외부에 반환할 task 객체를 어떻게 만들 것인가
코루틴이 처음 시작할 때 즉시 실행할 것인가
처음부터 중단할 것인가
최종 결과를 어디에 저장할 것인가
예외를 어떻게 저장할 것인가
코루틴이 끝날 때 중단할 것인가
자동으로 종료할 것인가
```

대표적으로 다음 멤버 함수들이 사용된다.

```cpp
struct promise_type {
    task get_return_object();

    std::suspend_always initial_suspend();
    std::suspend_always final_suspend() noexcept;

    void return_value(int value);
    void unhandled_exception();
};
```

각 함수의 역할을 하나씩 살펴보자.

## 7. `get_return_object()`

코루틴 함수를 호출하면 호출자에게 어떤 객체를 반환해야 한다.

예를 들어:

```cpp
task<int> result = calculate();
```

호출자는 코루틴 프레임 자체를 직접 받지 않는다.

대신 일반적으로 코루틴을 소유하거나 참조하는 `task<int>` 객체를 받는다.

이 객체를 만드는 것이 `get_return_object()`다.

```cpp
task get_return_object() {
    return task{
        std::coroutine_handle<promise_type>::from_promise(*this)
    };
}
```

여기서 중요한 객체가 등장한다.

```cpp
std::coroutine_handle<promise_type>
```

`coroutine_handle`은 코루틴 프레임을 가리키고 제어하기 위한 가벼운 핸들이다.

개념적으로는 포인터와 비슷한 역할을 하지만, 코루틴을 위한 전용 연산을 제공한다.

```text
resume()
done()
destroy()
promise()
```

따라서 `task<T>`는 보통 내부에 `coroutine_handle`을 보관한다.

```cpp
template <typename T>
class task {
private:
    std::coroutine_handle<promise_type> handle_;
};
```

구조를 정리하면 다음과 같다.

```text
호출자
  ↓
task<T>
  ↓ 보유
coroutine_handle
  ↓ 가리킴
coroutine frame
  ├─ promise_type
  ├─ 지역변수
  ├─ 상태
  └─ awaiter
```

## 8. `coroutine_handle`

`std::coroutine_handle`은 중단된 코루틴을 제어하는 핵심 객체다.

대표적인 사용법은 다음과 같다.

```cpp
handle.resume();
```

중단된 코루틴을 재개한다.

```cpp
if (handle.done()) {
    // 코루틴이 최종 중단 지점에 도달했는지 확인
}
```

```cpp
handle.destroy();
```

코루틴 프레임을 파괴한다.

```cpp
auto& promise = handle.promise();
```

코루틴 프레임 내부의 promise 객체에 접근한다.

C++의 함수 포인터와 비교하면 `coroutine_handle`은 단순히 실행할 함수 주소만 가진 것이 아니다.

개념적으로는 다음 두 정보에 연결된다.

```text
코루틴 실행 상태가 들어 있는 프레임
그 프레임을 재개하는 컴파일러 생성 코드
```

단순화하면 다음과 같이 생각할 수 있다.

```cpp
struct CoroutineHandle {
    void* frame_address;

    void resume();
    void destroy();
};
```

실제 구현은 표준 라이브러리와 ABI에 따라 달라지지만, 정신 모델로는 충분하다.

## 9. `initial_suspend()`

코루틴 프레임이 만들어진 뒤 함수 본문을 즉시 실행할지 여부는 `initial_suspend()`가 결정한다.

```cpp
std::suspend_always initial_suspend() noexcept {
    return {};
}
```

`std::suspend_always`를 반환하면 코루틴은 함수 본문을 실행하기 전에 중단된다.

```text
calculate() 호출
→ 코루틴 프레임 생성
→ task 객체 반환
→ 함수 본문은 아직 실행하지 않음
```

이런 방식을 흔히 **lazy coroutine**이라고 부른다.

실제 실행은 누군가 `resume()`하거나 해당 task를 `co_await`할 때 시작될 수 있다.

반대로 다음처럼 작성하면:

```cpp
std::suspend_never initial_suspend() noexcept {
    return {};
}
```

코루틴은 생성 직후 함수 본문 실행을 시작한다.

```text
calculate() 호출
→ 코루틴 프레임 생성
→ 함수 본문 즉시 실행
→ 첫 중단 지점까지 진행
```

이를 **eager coroutine**이라고 볼 수 있다.

즉, `initial_suspend()`는 코루틴 시작 정책을 결정한다.

```text
suspend_always
호출만으로는 실행하지 않음

suspend_never
호출 즉시 실행 시작
```

## 10. `final_suspend()`

코루틴이 `co_return`에 도달하거나 함수 본문 끝까지 실행되면 최종 단계로 들어간다.

이때 `final_suspend()`가 호출된다.

```cpp
std::suspend_always final_suspend() noexcept {
    return {};
}
```

최종 중단을 하는 이유는 코루틴이 끝난 뒤에도 호출자가 결과를 읽거나 프레임을 직접 파괴해야 할 수 있기 때문이다.

```text
코루틴 본문 완료
→ 결과 저장
→ final_suspend
→ 프레임은 아직 존재
→ 호출자가 결과 확인
→ destroy()
```

`final_suspend()`에서 중단하지 않고 프레임이 즉시 사라지게 만들 수도 있지만, 수명 관리가 매우 까다로워진다.

특히 외부 `task` 객체가 아직 코루틴 핸들을 가지고 있다면, 이미 파괴된 프레임을 가리키는 문제가 발생할 수 있다.

따라서 많은 간단한 task 구현은 `final_suspend()`에서 `std::suspend_always`를 사용하고, task 소멸자가 프레임을 파괴하도록 만든다.

```cpp
~task() {
    if (handle_) {
        handle_.destroy();
    }
}
```

여기에는 중요한 C++ 객체 수명 문제가 있다.

> `resume()`과 `destroy()`는 서로 다른 동작이며, 완료된 코루틴 프레임도 자동으로 항상 파괴되는 것은 아니다.

---

## 11. `co_return`

코루틴에서 값을 반환할 때는 일반 `return` 대신 `co_return`을 사용한다.

```cpp
co_return value * 2;
```

이 코드는 개념적으로 promise 객체의 `return_value()`를 호출한다.

```cpp
promise.return_value(value * 2);
```

예를 들어:

```cpp
struct promise_type {
    std::optional<int> result;

    void return_value(int value) {
        result = value;
    }
};
```

반환값이 없는 코루틴이라면 `return_void()`를 사용한다.

```cpp
struct promise_type {
    void return_void() {}
};
```

```cpp
task<void> run() {
    co_return;
}
```

대략 다음처럼 연결된다.

```text
co_return expression
→ promise.return_value(expression)

co_return
→ promise.return_void()
```

`co_return`이 실행되면 결과가 promise에 저장되고, 코루틴은 최종 중단 단계로 이동한다.

---

## 12. 예외는 어떻게 전달되는가

일반 함수에서 처리되지 않은 예외는 호출 스택을 따라 전파된다.

하지만 코루틴은 호출자와 동일한 연속 호출 스택 위에서 계속 실행되는 것이 아니다.

중간에 중단되고 나중에 다른 시점에 재개될 수 있다.

따라서 코루틴 내부에서 처리되지 않은 예외를 저장하는 방법이 필요하다.

이 역할을 하는 것이 `unhandled_exception()`이다.

```cpp
struct promise_type {
    std::exception_ptr exception;

    void unhandled_exception() {
        exception = std::current_exception();
    }
};
```

나중에 task 결과를 꺼낼 때 예외가 있다면 다시 던질 수 있다.

```cpp
int result() {
    if (promise.exception) {
        std::rethrow_exception(promise.exception);
    }

    return *promise.value;
}
```

즉, 코루틴의 예외 전달은 개념적으로 다음과 같다.

```text
코루틴 내부 예외 발생
→ promise에 exception_ptr 저장
→ 코루틴 완료
→ 호출자가 결과를 받을 때 예외 재발생
```

Python이나 JavaScript에서 `await` 시점에 비동기 예외가 다시 발생하는 구조와 비슷하다.

---

## 13. `co_await`는 무엇으로 변환되는가

이제 가장 중요한 `co_await`를 살펴보자.

```cpp
int value = co_await async_operation();
```

이 한 줄에는 다음 과정이 숨어 있다.

```text
awaitable 표현식 계산
awaiter 획득
결과 준비 여부 확인
필요하면 현재 코루틴 중단
나중에 코루틴 재개
결과 획득
```

핵심은 `co_await`가 **awaiter**라는 객체를 통해 동작한다는 것이다.

awaiter는 보통 다음 세 메서드를 제공한다.

```cpp
bool await_ready();

void await_suspend(std::coroutine_handle<> handle);

T await_resume();
```

각각의 역할은 다음과 같다.

```text
await_ready()
결과가 이미 준비되었는지 확인

await_suspend(handle)
아직 준비되지 않았다면 현재 코루틴을 중단

await_resume()
코루틴이 계속 실행될 때 결과를 반환
```

이 세 메서드가 C++ 코루틴 중단과 재개의 핵심 프로토콜이다.

---

## 14. Awaitable과 Awaiter는 무엇이 다른가

두 용어가 자주 함께 나오기 때문에 구분할 필요가 있다.

**Awaitable**은 `co_await` 뒤에 놓을 수 있는 표현식 또는 객체다.

```cpp
co_await async_operation();
```

여기서 `async_operation()`의 반환 객체가 awaitable일 수 있다.

**Awaiter**는 실제로 다음 메서드를 제공하는 객체다.

```cpp
await_ready()
await_suspend()
await_resume()
```

어떤 객체는 자신이 곧 awaiter일 수 있다.

```cpp
struct TimerAwaiter {
    bool await_ready() const noexcept;
    void await_suspend(std::coroutine_handle<>);
    void await_resume() const noexcept;
};
```

```cpp
co_await TimerAwaiter{};
```

반면 awaitable이 별도의 awaiter를 반환할 수도 있다.

```cpp
struct Timer {
    TimerAwaiter operator co_await() const {
        return TimerAwaiter{};
    }
};
```

```cpp
co_await Timer{};
```

이 경우 구조는 다음과 같다.

```text
Timer
→ operator co_await()
→ TimerAwaiter
→ await_ready / await_suspend / await_resume
```

즉:

```text
Awaitable
co_await가 적용될 수 있는 객체

Awaiter
실제 중단과 재개 프로토콜을 구현하는 객체
```

---

## 15. `await_ready()`

첫 번째 단계는 결과가 이미 준비되었는지 확인하는 것이다.

```cpp
bool await_ready() const noexcept {
    return completed;
}
```

`true`를 반환하면 코루틴은 중단되지 않는다.

```text
await_ready() == true
→ await_suspend() 호출 안 함
→ 즉시 await_resume()
→ 다음 코드 계속 실행
```

`false`를 반환하면 중단 과정으로 이동한다.

```text
await_ready() == false
→ await_suspend(handle)
→ 코루틴 중단 가능
```

이 때문에 `co_await`는 항상 코루틴을 중단시키는 문법이 아니다.

```cpp
int value = co_await operation;
```

정확한 의미는 다음과 같다.

> 결과가 아직 준비되지 않았다면 코루틴을 중단할 수 있는 지점이다.

예를 들어 이미 완료된 task나 캐시된 결과라면 중단하지 않고 즉시 실행을 계속할 수 있다.

---

## 16. `await_suspend()`

결과가 준비되지 않았다면 컴파일러는 현재 코루틴의 핸들을 `await_suspend()`에 전달한다.

```cpp
void await_suspend(std::coroutine_handle<> handle) {
    saved_handle = handle;
    start_async_operation();
}
```

여기서 전달받은 `handle`은 현재 실행 중인 코루틴을 나중에 재개하기 위한 정보다.

비동기 작업이 완료되면 다음과 같이 재개할 수 있다.

```cpp
saved_handle.resume();
```

실제 비동기 I/O에서는 다음과 같은 구조가 될 수 있다.

```text
현재 코루틴 핸들 저장
→ 소켓 읽기 작업 등록
→ 현재 코루틴 중단
→ 운영체제가 읽기 완료 통지
→ 이벤트 루프가 저장된 핸들 resume
```

단순화한 awaiter 예시는 다음과 같다.

```cpp
struct AsyncOperationAwaiter {
    std::coroutine_handle<> continuation;
    Result result;

    bool await_ready() const noexcept {
        return false;
    }

    void await_suspend(std::coroutine_handle<> handle) {
        continuation = handle;

        start_operation([this](Result value) {
            result = std::move(value);
            continuation.resume();
        });
    }

    Result await_resume() {
        return std::move(result);
    }
};
```

여기서 콜백 안에서 `resume()`을 호출한다.

즉, async/await가 콜백을 완전히 없앤 것은 아니다.

```text
기존 콜백
작업 완료 후 실행할 함수를 직접 전달

코루틴
작업 완료 후 재개할 coroutine_handle을 저장
```

코루틴은 continuation을 함수 객체 대신 코루틴 핸들로 표현할 수 있게 만든다.

---

## 17. `await_suspend()`의 반환 타입

`await_suspend()`는 반드시 `void`만 반환하는 것은 아니다.

대표적으로 다음 형태가 가능하다.

```cpp
void await_suspend(std::coroutine_handle<>);
```

이 경우 현재 코루틴은 중단된다.

```cpp
bool await_suspend(std::coroutine_handle<>);
```

`true`를 반환하면 중단하고, `false`를 반환하면 중단하지 않고 계속 실행할 수 있다.

```cpp
std::coroutine_handle<>
await_suspend(std::coroutine_handle<>);
```

다른 코루틴 핸들을 반환하면, 현재 코루틴에서 반환된 코루틴으로 직접 실행을 넘길 수 있다.

이 구조는 코루틴 간 연결에서 중요하다.

```text
부모 코루틴
→ 자식 코루틴을 co_await
→ 부모 중단
→ 자식 실행
→ 자식 완료
→ 부모 재개
```

다른 핸들을 반환하는 방식은 불필요한 스케줄러 왕복을 줄이는 **symmetric transfer**와 연결된다.

처음 학습할 때는 다음 정도만 기억해도 충분하다.

```text
await_suspend는 현재 코루틴 핸들을 받는다.
그리고 현재 코루틴을 언제, 어떻게 다시 실행할지 등록한다.
```

---

## 18. `await_resume()`

코루틴이 재개되거나 중단할 필요가 없었던 경우, `await_resume()`이 호출된다.

```cpp
Result await_resume() {
    return result;
}
```

`co_await` 표현식 전체의 결과가 바로 `await_resume()`의 반환값이다.

```cpp
int value = co_await async_operation();
```

개념적으로는:

```cpp
int value = awaiter.await_resume();
```

와 연결된다.

오류가 발생했다면 `await_resume()`에서 예외를 던질 수도 있다.

```cpp
Result await_resume() {
    if (exception) {
        std::rethrow_exception(exception);
    }

    return std::move(result);
}
```

따라서 awaiter는 다음 세 단계의 책임을 나눠 가진다.

```text
await_ready
지금 바로 결과를 받을 수 있는가?

await_suspend
아직 안 된다면 현재 코루틴을 어떻게 대기시킬 것인가?

await_resume
완료 후 어떤 결과를 반환하거나 어떤 예외를 던질 것인가?
```

---

## 19. 간단한 수동 Awaiter 만들기

비동기 I/O 없이 중단과 재개만 관찰하는 간단한 awaiter를 만들어보자.

```cpp
#include <coroutine>
#include <iostream>

struct ManualAwaiter {
    std::coroutine_handle<> suspended_handle;

    bool await_ready() const noexcept {
        std::cout << "await_ready\n";
        return false;
    }

    void await_suspend(std::coroutine_handle<> handle) noexcept {
        std::cout << "await_suspend\n";
        suspended_handle = handle;
    }

    int await_resume() const noexcept {
        std::cout << "await_resume\n";
        return 42;
    }

    void resume() {
        if (suspended_handle) {
            suspended_handle.resume();
        }
    }
};
```

사용 코드는 다음과 같다.

```cpp
ManualAwaiter awaiter;

task<void> example() {
    std::cout << "before co_await\n";

    int value = co_await awaiter;

    std::cout << "after co_await: "
              << value << '\n';
}
```

실행 흐름은 개념적으로 다음과 같다.

```text
example 코루틴 실행
→ before co_await 출력
→ await_ready 호출
→ false
→ await_suspend 호출
→ 코루틴 중단

외부에서 awaiter.resume()

→ 코루틴 재개
→ await_resume 호출
→ 42 반환
→ value = 42
→ after co_await 출력
```

여기서 핵심은 `await_suspend()`가 다음 실행 코드를 직접 저장하는 것이 아니라, 현재 코루틴의 핸들을 저장한다는 점이다.

```cpp
suspended_handle = handle;
```

나중에:

```cpp
suspended_handle.resume();
```

를 호출하면 컴파일러가 저장해 둔 상태에 따라 `co_await` 이후부터 실행된다.

## 20. 최소한의 `task<T>`는 어떤 구조인가

C++ 표준에는 일반적인 `task<T>`가 없다.

따라서 코루틴 결과 타입을 직접 만들거나 라이브러리에서 제공하는 타입을 사용해야 한다.

아주 단순화한 `task<int>` 구조를 생각해보자.

```cpp
#include <coroutine>
#include <exception>
#include <optional>
#include <utility>

class IntTask {
public:
    struct promise_type;
    using handle_type =
        std::coroutine_handle<promise_type>;

    struct promise_type {
        std::optional<int> value;
        std::exception_ptr exception;

        IntTask get_return_object() {
            return IntTask{
                handle_type::from_promise(*this)
            };
        }

        std::suspend_always
        initial_suspend() noexcept {
            return {};
        }

        std::suspend_always
        final_suspend() noexcept {
            return {};
        }

        void return_value(int result) noexcept {
            value = result;
        }

        void unhandled_exception() noexcept {
            exception = std::current_exception();
        }
    };

    explicit IntTask(handle_type handle)
        : handle_(handle) {}

    IntTask(const IntTask&) = delete;

    IntTask(IntTask&& other) noexcept
        : handle_(std::exchange(other.handle_, {})) {}

    ~IntTask() {
        if (handle_) {
            handle_.destroy();
        }
    }

    void resume() {
        if (handle_ && !handle_.done()) {
            handle_.resume();
        }
    }

    bool done() const {
        return !handle_ || handle_.done();
    }

    int result() {
        auto& promise = handle_.promise();

        if (promise.exception) {
            std::rethrow_exception(promise.exception);
        }

        return *promise.value;
    }

private:
    handle_type handle_;
};
```

이 타입을 사용하는 코루틴은 다음과 같다.

```cpp
IntTask calculate() {
    co_return 21 * 2;
}
```

```cpp
int main() {
    IntTask task = calculate();

    task.resume();

    if (task.done()) {
        std::cout << task.result() << '\n';
    }
}
```

이 예제는 비동기 I/O를 수행하지 않는다.

단지 다음 구조를 보여준다.

```text
코루틴 함수 호출
→ 프레임 생성
→ IntTask 반환
→ task가 coroutine_handle 보유
→ resume()으로 함수 본문 실행
→ co_return 결과를 promise에 저장
→ final_suspend
→ result()로 결과 조회
→ task 소멸 시 프레임 destroy
```

실무용 `task<T>`는 이보다 훨씬 더 많은 기능이 필요하다.

```text
다른 코루틴에서 co_await 가능
부모 코루틴 continuation 저장
자식 완료 후 부모 자동 재개
취소
동기화
스레드 안전성
스케줄러 연동
결과 수명 관리
```

## 21. `task<T>`를 `co_await`하려면 무엇이 필요한가

다음과 같은 코드를 작성하고 싶다고 하자.

```cpp
task<int> child() {
    co_return 10;
}

task<int> parent() {
    int value = co_await child();
    co_return value * 2;
}
```

`child()`가 반환하는 `task<int>`는 awaitable이어야 한다.

즉, `task<int>` 자체가 awaiter를 제공하거나 `operator co_await()`를 통해 awaiter를 반환해야 한다.

개념적인 구조는 다음과 같다.

```cpp
class task {
public:
    struct awaiter {
        handle_type child_handle;

        bool await_ready() const noexcept {
            return child_handle.done();
        }

        std::coroutine_handle<>
        await_suspend(
            std::coroutine_handle<> parent
        ) noexcept {
            child_handle.promise().continuation =
                parent;

            return child_handle;
        }

        T await_resume() {
            return child_handle.promise().result();
        }
    };

    awaiter operator co_await() {
        return awaiter{handle_};
    }
};
```

흐름은 다음과 같다.

```text
parent 실행
→ child() 호출
→ child task 생성
→ parent가 child task를 co_await

child가 완료되지 않음
→ parent handle을 child promise에 continuation으로 저장
→ parent 중단
→ child 실행

child co_return
→ child final_suspend
→ 저장된 parent handle 재개
→ parent의 await_resume
→ child 결과 획득
→ parent 계속 실행
```

여기서 `continuation`은 이전 글에서 설명한 "작업 완료 후 이어서 실행할 위치"다.

C++에서는 이를 `coroutine_handle`로 직접 표현할 수 있다.

---

## 22. `final_suspend()`와 부모 코루틴 재개

자식 코루틴이 완료됐을 때 부모 코루틴을 다시 실행하려면 `final_suspend()`가 중요한 역할을 한다.

예를 들어 promise에 부모 코루틴 핸들을 저장한다고 하자.

```cpp
std::coroutine_handle<> continuation;
```

자식이 끝나면 final awaiter가 부모 핸들을 반환할 수 있다.

```cpp
struct FinalAwaiter {
    bool await_ready() const noexcept {
        return false;
    }

    std::coroutine_handle<>
    await_suspend(
        std::coroutine_handle<promise_type> handle
    ) const noexcept {
        auto continuation =
            handle.promise().continuation;

        if (continuation) {
            return continuation;
        }

        return std::noop_coroutine();
    }

    void await_resume() const noexcept {}
};
```

```cpp
FinalAwaiter final_suspend() noexcept {
    return {};
}
```

이 구조에서는 자식 코루틴 완료 후 부모 코루틴으로 직접 실행이 넘어간다.

```text
parent
→ child
→ child 완료
→ parent 재개
```

스레드의 호출 스택에서는 함수 반환 주소가 자동으로 부모 함수의 다음 위치를 가리킨다.

코루틴에서는 실행이 분리되어 있으므로, 부모 코루틴의 핸들을 명시적으로 continuation으로 저장해야 한다.

이 차이가 코루틴의 비동기 호출 체인을 이해하는 핵심이다.

---

## 23. 일반 호출 스택과 코루틴 체인

일반 함수 호출은 하나의 연속된 호출 스택을 만든다.

```text
main()
└─ parent()
   └─ child()
      └─ operation()
```

`operation()`이 반환하면 자동으로 `child()`, `parent()`, `main()` 순서로 돌아간다.

반면 코루틴은 각 실행 상태가 별도의 프레임으로 존재할 수 있다.

```text
parent coroutine frame
    awaits
child coroutine frame
    awaits
operation awaiter
```

자식 코루틴이 중단되면 부모 코루틴 역시 결과를 받을 수 없기 때문에 중단될 수 있다.

하지만 실제 프레임들은 하나의 연속된 스택에 쌓여 있는 것이 아니라, 핸들과 continuation으로 연결된다.

```text
parent handle
    ↑ continuation
child promise
```

즉, 비동기 호출 체인은 일반 호출 스택 대신 다음 구조를 형성한다.

> 여러 코루틴 프레임이 `coroutine_handle`과 continuation으로 연결된 논리적인 호출 체인

---

## 24. 코루틴이 실행될 스레드는 누가 결정하는가

`co_await` 자체는 새로운 스레드를 만들지 않는다.

또한 `coroutine_handle.resume()`은 기본적으로 호출한 스레드에서 코루틴을 실행한다.

```cpp
handle.resume();
```

이 코드를 스레드 A가 호출하면, 코루틴은 스레드 A에서 재개된다.

```text
Thread A
→ handle.resume()
→ 코루틴 코드 실행
```

다른 스레드에서 재개하고 싶다면 스케줄러나 실행기에 핸들을 넘겨야 한다.

```cpp
void await_suspend(
    std::coroutine_handle<> handle
) {
    thread_pool.enqueue(handle);
}
```

스레드 풀의 작업 스레드가 나중에:

```cpp
handle.resume();
```

를 실행하면 코루틴은 해당 작업 스레드에서 재개된다.

따라서 C++ 코루틴은 실행 스레드에 대해 중립적이다.

```text
코루틴 언어 기능
중단과 재개 방법 제공

스케줄러
어느 스레드에서 언제 재개할지 결정
```

이 점은 JavaScript와 Python의 일반적인 async/await 사용 방식과 차이가 있다.

JavaScript와 Python에서는 이벤트 루프가 실행 정책을 상당 부분 제공한다.

C++에서는 사용 중인 라이브러리와 awaiter 구현이 재개 정책을 결정한다.

---

## 25. 스케줄러 전환 Awaiter

코루틴을 특정 실행기로 이동시키는 awaiter를 만들 수도 있다.

```cpp
struct ScheduleAwaiter {
    Executor& executor;

    bool await_ready() const noexcept {
        return false;
    }

    void await_suspend(
        std::coroutine_handle<> handle
    ) {
        executor.enqueue(handle);
    }

    void await_resume() const noexcept {}
};
```

사용은 다음과 같다.

```cpp
task<void> run_on_worker(Executor& executor) {
    co_await ScheduleAwaiter{executor};

    // 이 지점부터 executor가 선택한 스레드에서
    // 실행될 수 있다.
}
```

흐름은 다음과 같다.

```text
현재 스레드에서 코루틴 실행
→ co_await ScheduleAwaiter
→ 현재 코루틴 핸들을 executor queue에 등록
→ 현재 코루틴 중단
→ worker thread가 handle.resume()
→ worker thread에서 코루틴 재개
```

이처럼 `co_await`는 비동기 결과만 기다리는 문법이 아니다.

awaiter가 정의한 동작에 따라 다음과 같은 일도 할 수 있다.

```text
타이머 대기
네트워크 I/O 대기
스레드 풀 이동
락 획득 대기
다른 코루틴 완료 대기
사용자 정의 이벤트 대기
```

`co_await`의 의미는 고정되어 있지 않다.

> `co_await` 뒤에 놓인 awaiter가 무엇을 하도록 설계됐는지가 실제 동작을 결정한다.

## 26. 코루틴은 스레드보다 가볍다는 말의 의미

스레드는 운영체제가 관리하는 실행 단위다.

각 스레드는 일반적으로 자체 호출 스택과 스케줄링 상태를 가진다.

코루틴은 언어와 라이브러리가 관리하는 실행 상태다.

일반적으로 필요한 상태만 코루틴 프레임에 저장한다.

```text
스레드
운영체제 스케줄링 단위
자체 스택
선점 가능
비교적 높은 생성 및 전환 비용

코루틴
사용자 공간 실행 단위
코루틴 프레임
명시적 중단 지점
일반적으로 더 작은 상태와 낮은 전환 비용
```

다만 "코루틴은 무조건 빠르다"라고 단정하면 안 된다.

성능은 다음 요소에 따라 달라진다.

```text
코루틴 프레임 할당
awaiter 구현
스케줄러 큐
스레드 간 이동
동기화 비용
캐시 지역성
컴파일러 최적화
```

코루틴의 가장 직접적인 장점은 단순히 속도가 아니다.

> 비동기 실행의 continuation을 순차적인 코드 형태로 표현하면서, 실행 상태를 컴파일러가 관리하게 한다.

## 27. 코루틴과 객체 수명

C++ 코루틴에서 가장 조심해야 할 부분 중 하나가 객체 수명이다.

다음 코드를 보자.

```cpp
task<void> process(const std::string& value) {
    co_await async_operation();
    std::cout << value;
}
```

`value`는 참조다.

코루틴이 중단된 동안 원본 문자열이 파괴되면 재개 후 댕글링 참조를 사용할 수 있다.

```cpp
task<void> start() {
    std::string text = "hello";

    auto work = process(text);

    // text가 먼저 파괴될 가능성이 있다면 위험
}
```

일반 함수라면 호출이 끝나기 전에 참조 대상이 살아 있는 경우가 많다.

하지만 코루틴은 호출자의 범위를 벗어난 뒤에도 실행될 수 있다.

따라서 다음 항목을 반드시 생각해야 한다.

```text
코루틴 프레임 안에 값이 복사되는가
참조나 포인터만 저장되는가
await 동안 원본 객체가 살아 있는가
this 객체가 코루틴 완료 전까지 살아 있는가
task를 누가 소유하는가
코루틴 취소 시 프레임은 언제 파괴되는가
```

예를 들어 멤버 코루틴도 위험할 수 있다.

```cpp
class Worker {
public:
    task<void> run() {
        co_await async_operation();
        use(member_);
    }

private:
    int member_;
};
```

`run()`이 중단된 동안 `Worker` 객체가 파괴되면 코루틴 프레임에 저장된 `this` 포인터가 무효가 된다.

코루틴 프레임이 존재한다고 해서 참조하는 외부 객체까지 자동으로 살아 있는 것은 아니다.

## 28. RAII와 코루틴

C++에서는 객체가 스코프를 벗어날 때 소멸자가 실행되는 RAII가 중요하다.

코루틴에서도 지역 객체는 코루틴 프레임에 저장될 수 있으며, 코루틴 프레임이 파괴될 때 소멸한다.

```cpp
task<void> example() {
    Resource resource;

    co_await async_operation();

    use(resource);
}
```

`resource`는 `co_await` 동안에도 살아 있어야 하므로 프레임에 포함될 수 있다.

정상 완료되거나 프레임이 `destroy()`되면 `resource`의 소멸자가 실행된다.

하지만 코루틴 프레임이 누수되면 내부 RAII 객체도 파괴되지 않는다.

따라서 task 소유권 관리가 매우 중요하다.

```cpp
~task() {
    if (handle_) {
        handle_.destroy();
    }
}
```

또한 코루틴이 중단된 상태에서 프레임을 파괴하면, 함수 본문의 나머지 코드는 실행하지 않고 프레임 내부 객체의 소멸만 진행된다.

```text
코루틴 중단
→ destroy()
→ 살아 있는 지역 객체 소멸
→ 프레임 해제
→ 코루틴 본문 재개 안 됨
```

## 29. 코루틴과 락을 함께 사용할 때

코루틴은 일반 함수보다 긴 시간 동안 중단될 수 있다.

따라서 락을 보유한 상태에서 `co_await`하는 것은 특히 위험하다.

```cpp
task<void> unsafe() {
    std::lock_guard lock(mutex);

    co_await async_operation();

    update_shared_state();
}
```

이 코드에서는 `async_operation()`이 완료될 때까지 락이 유지될 수 있다.

```text
mutex 획득
→ co_await에서 코루틴 중단
→ 장시간 락 유지
→ 다른 작업이 mutex 획득 불가
```

더 나쁜 경우에는 재개를 위해 같은 락이 필요한 작업과 교착 상태가 발생할 수 있다.

일반적인 원칙은 다음과 같다.

> 락을 보유한 채로 코루틴이 중단될 수 있는 `co_await`를 수행하지 않는다.

가능하면 다음처럼 락의 범위를 중단 지점 이전에 끝내야 한다.

```cpp
task<void> safer() {
    Data snapshot;

    {
        std::lock_guard lock(mutex);
        snapshot = shared_data;
    }

    auto result = co_await async_operation(snapshot);

    {
        std::lock_guard lock(mutex);
        apply(result);
    }
}
```

## 30. `co_await`와 동기화 지점

협력적 스케줄링에서는 코루틴이 `co_await` 지점에서 다른 작업에 실행권을 넘길 수 있다.

따라서 `co_await` 전후에는 다른 작업이 공유 상태를 변경했을 가능성을 고려해야 한다.

```cpp
if (state.ready()) {
    co_await async_operation();
    use(state.value());
}
```

`co_await` 전에 `state.ready()`가 참이었더라도, 재개 후 상태가 그대로라는 보장은 없다.

```text
state.ready() 확인
→ co_await에서 중단
→ 다른 코루틴이 state 변경
→ 현재 코루틴 재개
→ 이전 조건이 더 이상 유효하지 않을 수 있음
```

스레드 컨텍스트 스위치는 어느 시점에서든 발생할 수 있지만, 단일 스레드 코루틴에서는 상태 변화 지점을 `co_await` 중심으로 추적하기 쉬운 장점이 있다.

그렇다고 데이터 경쟁이 자동으로 사라지는 것은 아니다.

코루틴이 여러 스레드에서 재개될 수 있거나 공유 상태를 사용한다면 기존 동기화 문제는 그대로 존재한다.

## 31. C++20 코루틴이 제공하는 것과 제공하지 않는 것

이제 C++20의 역할을 명확하게 구분해보자.

### C++20이 제공하는 것

```text
코루틴 함수 변환 규칙
코루틴 프레임
co_await, co_yield, co_return
promise_type 연결 규칙
coroutine_handle
suspend_always, suspend_never
awaiter 프로토콜
```

### C++20이 제공하지 않는 것

```text
표준 task<T>
표준 이벤트 루프
표준 네트워크 비동기 실행 모델
표준 코루틴 스케줄러

```

따라서 실제 비동기 프로그램에서는 라이브러리가 필요하다.

예를 들면:

```text
Boost.Asio

플랫폼별 비동기 I/O 라이브러리
프로젝트 자체 task 및 executor 구현
```

라이브러리는 C++20 코루틴 언어 기능 위에 다음을 제공한다.

```text
실제 네트워크 작업
타이머
스케줄링
task 수명 관리
오류 전달
취소
여러 코루틴 조합
```

## 32. JavaScript와 Python보다 C++ 코루틴이 어려운 이유

JavaScript에서는 일반적으로 다음 코드만 작성하면 된다.

```javascript
const result = await fetch(url);
```

Promise와 이벤트 루프가 이미 런타임에 통합되어 있다.

Python에서도 다음처럼 작성한다.

```python
result = await session.call_tool(...)
```

`asyncio`가 Task, Future, Event Loop를 제공한다.

C++에서는 `co_await` 문법은 제공하지만, 그 뒤의 실행 모델은 라이브러리에 따라 다르다.

```cpp
auto result = co_await operation;
```

이 한 줄을 이해하려면 다음 질문이 필요하다.

```text
operation은 어떤 타입인가
operator co_await를 제공하는가
awaiter는 무엇인가
await_suspend에서 핸들을 어디에 저장하는가
누가 resume을 호출하는가
어느 스레드에서 재개되는가
결과는 어디에 저장되는가
task 프레임은 누가 파괴하는가
```

C++의 장점은 이 모든 정책을 세밀하게 제어할 수 있다는 것이다.

단점은 같은 문법이라도 라이브러리에 따라 수명, 실행 스레드, 취소, 예외 처리 방식이 달라질 수 있다는 것이다.

## 33. 전체 실행 흐름 다시 보기

다음 코드를 기준으로 전체 흐름을 정리해보자.

```cpp
task<int> calculate() {
    int value = co_await async_operation();
    co_return value * 2;
}
```

호출자가:

```cpp
task<int> result = calculate();
```

를 실행하면 개념적으로 다음 일이 일어난다.

```text
1. 코루틴 프레임 메모리 확보
```

## 34. 가장 중요한 정신 모델

C++ 코루틴을 볼 때 다음처럼 생각하면 된다.

```cpp
auto value = co_await operation;
```

이 코드는 단순히 "operation을 기다린다"는 뜻이 아니다.

개념적으로는 다음과 같다.

```text
operation에서 awaiter를 얻는다.

결과가 이미 있으면:
    즉시 결과를 받아 계속 실행한다.

결과가 아직 없으면:
    현재 coroutine_handle을 awaiter에 전달한다.
    현재 코루틴의 실행 위치와 지역변수를 프레임에 남긴다.
    호출자에게 실행권을 돌려준다.

작업이 완료되면:
    누군가 저장된 handle.resume()을 호출한다.
    코루틴은 await_resume()으로 결과를 얻는다.
    co_await 다음 줄부터 계속 실행한다.
```

이를 C++ 요소로 압축하면 다음과 같다.

```text
Coroutine Frame
실행 상태와 지역변수 저장

promise_type
코루틴의 반환, 예외, 시작과 종료 정책 정의

coroutine_handle
중단된 코루틴을 참조하고 재개하거나 파괴

Awaiter
언제 중단하고 언제 재개할지 정의

Scheduler 또는 I/O Runtime
저장된 handle을 언제 어느 스레드에서 resume할지 결정
```

## 마무리

C++20 Coroutine은 `co_await` 하나로 비동기를 완성해 주는 기능이 아니다.

C++20이 제공하는 것은 다음과 같은 저수준 메커니즘이다.

```text
함수 실행 상태를 코루틴 프레임에 보존한다.
현재 코루틴을 중단한다.
coroutine_handle을 통해 나중에 재개한다.
promise_type으로 반환값과 수명을 제어한다.
awaiter로 중단과 재개의 조건을 정의한다.
```

따라서 C++ 코루틴의 핵심은 다음 한 문장으로 정리할 수 있다.

> `co_await`는 현재 스레드를 자동으로 멈추는 문법이 아니라, awaiter에게 현재 코루틴의 continuation인 `coroutine_handle`을 넘겨 중단과 재개를 협의하는 문법이다.

이 구조를 이해하면 Python과 JavaScript의 `await`도 더 구체적으로 보인다.

```text
C++20
awaiter가 coroutine_handle을 저장하고 나중에 resume

Python
Future가 Task의 재개를 이벤트 루프에 예약

JavaScript
Promise 완료 후 continuation을 microtask queue에 예약
```

표현과 런타임은 다르지만 공통 원리는 같다.

```text
실행 상태 저장
→ continuation 등록
→ 현재 실행 중단
→ 결과 완료
→ continuation 재개
```

다음 글에서는 Python의 `async/await`를 살펴본다.

특히 다음 요소가 실제로 어떻게 연결되는지를 MCP 호출 예제로 확인한다.

```text

Awaitable
Task

Event Loop
asyncio.create_task

```

C++에서는 개발자가 직접 보던 `coroutine_handle`, awaiter, 스케줄링 구조가 Python에서는 어떤 객체와 런타임 동작 뒤에 숨겨져 있는지 비교해볼 것이다.
