## 문제 설명

숫자로만 이루어진 문자열 `s`와 쿼리 배열 `queries`가 주어진다. 각 쿼리 `[l, r]`에 대해 부분 문자열 `s[l..r]`을 대상으로 다음을 계산해야 한다.

- 부분 문자열의 자릿수 중 **0이 아닌 자릿수만** 순서대로 이어 붙여 정수 `x`를 만든다.
- 부분 문자열을 이루는 **모든 자릿수의 합** `sum`을 구한다.
- `x * sum`을 `10^9 + 7`로 나눈 나머지를 답으로 한다.

이전 문제(3754번, Concatenate Non-Zero Digits and Multiply by Sum I)가 문자열 전체에 대해 한 번만 계산하면 됐다면, 이 문제는 **여러 개의 구간 쿼리**가 주어지고 문자열 길이와 쿼리 개수가 커질 수 있어 매 쿼리마다 처음부터 순회하면 시간 초과가 발생한다.

핵심 관찰은 다음과 같다.

- `sum`은 일반적인 구간 합이므로 **prefix sum**으로 $O(1)$에 구할 수 있다.
- `x`는 단순 합이 아니라 "0이 아닌 자릿수를 순서대로 이어 붙인 값"이라 자리 이동(`* 10^k`)이 필요하다. 두 구간을 합칠 때 왼쪽 값에 오른쪽 구간의 0이 아닌 자릿수 개수만큼 자리를 밀어주고 오른쪽 값을 더하면 된다: `merge(L, R).val = L.val * 10^(R.cnt) + R.val`.
- 이 병합 규칙은 **결합법칙이 성립**하므로 세그먼트 트리로 구간 병합이 가능하고, 동시에 prefix 배열로도 표현 가능하다 (구간 `[l, r]`의 값은 `prefix[r+1]`에서 `prefix[l]`을 `10^(구간 내 0이 아닌 자릿수 개수)`만큼 밀어서 뺀 값과 같다).

<br/>

## 내 아이디어

이어 붙인 값 `x`를 만드는 연산은 "0을 건너뛰고 자리를 이동시켜 붙이는" 비선형적인 연산이라 세그먼트 트리처럼 구간을 재귀적으로 병합하는 구조가 가장 먼저 떠올랐다. 반면 `sum`은 단순 합이라 prefix sum으로 충분하다고 판단했다.

- 세그먼트 트리의 각 노드에 `(val, sum, cnt)` 세 값을 저장한다. `val`은 해당 구간에서 0이 아닌 자릿수를 이어 붙인 값, `sum`은 자릿수 합, `cnt`는 0이 아닌 자릿수 개수다.
- 두 자식 노드를 합칠 때 `val`은 `left.val * 10^(right.cnt) + right.val`로 자리를 이동시켜 이어 붙이고, `sum`과 `cnt`는 단순히 더한다.
- 리프 노드는 문자 하나에 대응하며, 0이면 `(0, 0, 0)`, 0이 아니면 `(d, d, 1)`이다.
- 쿼리마다 구간 `[l, r]`을 세그먼트 트리에서 질의해 `(val, sum)`을 얻고 `val * sum % MOD`를 반환한다.

이후 같은 병합 규칙이 **구간 트리 없이 prefix 배열만으로도** 표현될 수 있다는 것을 깨달았다. `val[i]`를 `s[0..i-1]`까지의 이어 붙인 값, `cnt[i]`를 그 구간의 0이 아닌 자릿수 개수라 하면, 구간 `[l, r]`의 `x`는 `val[r+1] - val[l] * 10^(cnt[r+1]-cnt[l])`로 계산할 수 있다. 이렇게 하면 각 쿼리를 $O(1)$에 처리할 수 있어 세그먼트 트리보다 더 간단하고 빠르다.

<br/>

## 내 아이디어로 구현된 코드

```cpp
class Solution {
public:
    static const long long MOD = 1'000'000'007;

    struct Node {
        long long val; // non-zero digits concatenated value
        long long sum; // digit sum
        int cnt;       // number of non-zero digits

        Node(long long val = 0, long long sum = 0, int cnt = 0)
            : val(val), sum(sum), cnt(cnt) {}
    };

    string s;
    vector<Node> tree;
    vector<long long> pow10;

    Node mergeNode(const Node& left, const Node& right) {
        Node res;
        res.cnt = left.cnt + right.cnt;
        res.sum = left.sum + right.sum;

        res.val = (left.val * pow10[right.cnt] + right.val) % MOD;

        return res;
    }

    void build(int node, int start, int end) {
        if (start == end) {
            int d = s[start] - '0';

            if (d == 0) {
                tree[node] = Node(0, 0, 0);
            } else {
                tree[node] = Node(d, d, 1);
            }

            return;
        }

        int mid = (start + end) / 2;

        build(node * 2, start, mid);
        build(node * 2 + 1, mid + 1, end);

        tree[node] = mergeNode(tree[node * 2], tree[node * 2 + 1]);
    }

    Node query(int node, int start, int end, int left, int right) {
        if (right < start || end < left) {
            return Node(0, 0, 0);
        }

        if (left <= start && end <= right) {
            return tree[node];
        }

        int mid = (start + end) / 2;

        Node leftNode = query(node * 2, start, mid, left, right);
        Node rightNode = query(node * 2 + 1, mid + 1, end, left, right);

        return mergeNode(leftNode, rightNode);
    }

    vector<int> sumAndMultiply(string s, vector<vector<int>>& queries) {
        this->s = s;

        int n = s.size();

        pow10.assign(n + 1, 1);
        for (int i = 1; i <= n; i++) {
            pow10[i] = pow10[i - 1] * 10 % MOD;
        }

        tree.assign(4 * n, Node());

        build(1, 0, n - 1);

        vector<int> ans;
        ans.reserve(queries.size());

        for (auto& q : queries) {
            int l = q[0];
            int r = q[1];

            Node cur = query(1, 0, n - 1, l, r);

            long long result = cur.val * cur.sum % MOD;
            ans.push_back(result);
        }

        return ans;
    }
};
```

- **Time Complexity:** $O((n + m) \log n)$ (트리 구성 $O(n)$, 쿼리당 $O(\log n)$, 쿼리 $m$개)
- **Space Complexity:** $O(n)$

### 동작 원리

1. `build`로 세그먼트 트리를 구성한다. 리프는 문자 하나에 대응하고, 내부 노드는 두 자식을 `mergeNode`로 합친 결과를 저장한다.
2. `mergeNode`는 왼쪽 구간의 이어붙인 값에 오른쪽 구간의 0이 아닌 자릿수 개수(`right.cnt`)만큼 `10`의 거듭제곱을 곱해 자리를 밀어준 뒤 오른쪽 값을 더한다. `sum`과 `cnt`는 그냥 더하면 된다.
3. 쿼리 `[l, r]`이 들어오면 `query`로 해당 구간에 대응하는 `Node`를 얻고, 그 안의 `val`(이어붙인 값)과 `sum`(자릿수 합)을 곱해 `MOD`로 나눈 나머지를 반환한다.
4. 트리 병합 규칙 자체가 결합법칙을 만족하므로, 구간을 여러 조각으로 쪼개 병합해도 항상 올바른 `(val, sum, cnt)`가 나온다.

## 풀이 1: Prefix 배열 (쿼리당 계산)

```cpp
class Solution {
public:
    vector<int> sumAndMultiply(string s, vector<vector<int>>& queries) {
        const long long MOD = 1'000'000'007;
        int n = s.size();

        vector<long long> pow10(n + 1, 1);
        for (int i = 1; i <= n; i++) {
            pow10[i] = pow10[i - 1] * 10 % MOD;
        }

        vector<int> sumD(n + 1, 0);
        vector<int> cntN0(n + 1, 0);
        vector<long long> val(n + 1, 0);

        for (int i = 1; i <= n; i++) {
            int d = s[i - 1] - '0';

            sumD[i] = sumD[i - 1] + d;
            cntN0[i] = cntN0[i - 1] + (d != 0);

            if (d != 0) {
                val[i] = (val[i - 1] * 10 + d) % MOD;
            } else {
                val[i] = val[i - 1];
            }
        }

        vector<int> ans;
        ans.reserve(queries.size());

        for (auto& q : queries) {
            int l = q[0];
            int r = q[1];

            int k = cntN0[r + 1] - cntN0[l];
            long long sd = sumD[r + 1] - sumD[l];

            long long x = (val[r + 1] - val[l] * pow10[k] % MOD + MOD) % MOD;

            ans.push_back(x * sd % MOD);
        }

        return ans;
    }
};
```

- **Time Complexity:** $O(n + m)$ (전처리 $O(n)$, 쿼리당 $O(1)$)
- **Space Complexity:** $O(n)$

### 핵심 아이디어

세그먼트 트리의 병합 규칙 `left.val * 10^(right.cnt) + right.val`을 그대로 prefix 형태로 옮긴 것이다. `val[i]`는 `s[0..i-1]`까지 이어 붙인 값이므로, 구간 `[l, r]`의 값은 `val[r+1]`에서 `val[l]`을 구간 내 0이 아닌 자릿수 개수(`k`)만큼 자리를 밀어(`* 10^k`) 뺀 값과 같다.

세그먼트 트리처럼 재귀적으로 구간을 쪼개 병합할 필요 없이, prefix 배열 하나로 임의의 구간을 즉시 계산할 수 있어 쿼리당 $O(1)$로 줄어든다. 트리 구조와 재귀 호출이 없어 코드도 더 간결하다.

## 풀이 2: Prefix 배열 + 전역 사전 계산

```cpp
const int MOD = 1e9 + 7;
const int MAX_N = 100001;
long long pow10[MAX_N];

// init runs only once for all test cases
int init = []() {
    pow10[0] = 1;
    for (int i = 1; i < MAX_N; ++i) {
        pow10[i] = (pow10[i - 1] * 10) % MOD;
    }
    return 0;
}();

class Solution {
public:
    vector<int> sumAndMultiply(string s, vector<vector<int>>& queries) {
        int n = s.size();
        vector<int> sum(n + 1, 0);
        vector<long long> x(n + 1, 0);
        vector<int> cnt(n + 1, 0);
        for (int i = 0; i < n; ++i) {
            int d = s[i] - '0';
            sum[i + 1] = sum[i] + d;
            x[i + 1] = (d > 0) ? (x[i] * 10 + d) % MOD : x[i];
            cnt[i + 1] = cnt[i] + (d > 0);
        }
        int m = queries.size();
        vector<int> res(m, 0);
        for (int i = 0; i < m; ++i) {
            int l = queries[i][0];
            int r = queries[i][1] + 1;
            int length = cnt[r] - cnt[l];
            long long val_x = (x[r] - x[l] * pow10[length] % MOD + MOD) % MOD;
            long long val_sum = sum[r] - sum[l];
            res[i] = (val_x * val_sum) % MOD;
        }
        return res;
    }
};
```

- **Time Complexity:** $O(n + m)$ (`pow10`은 전역에서 한 번만 계산되므로 각 테스트 케이스 실행 시간에는 포함되지 않는다)
- **Space Complexity:** $O(n + \text{MAX\_N})$

### 핵심 아이디어

풀이 1과 로직은 동일하지만, `pow10` 배열을 함수 내부가 아니라 **전역 변수 + 즉시 실행 람다(`init`)**로 한 번만 계산해둔다. LeetCode 채점 환경은 하나의 프로세스에서 여러 테스트 케이스(또는 스트레스 테스트의 여러 호출)를 연속으로 실행하는 경우가 있는데, 이때 `pow10`을 매번 다시 계산하지 않고 재사용할 수 있어 실질적인 실행 시간을 아낄 수 있다.

다만 `MAX_N`을 문제의 최대 길이에 맞춰 미리 고정해야 하고, 여러 테스트 케이스 간에 전역 상태를 공유하는 방식이라 일반적인 실무 코드 스타일과는 다소 거리가 있다. 알고리즘 자체의 시간 복잡도는 풀이 1과 동일하다.

## 풀이 비교

| 항목 | 내 코드 (세그먼트 트리) | 풀이 1 (Prefix, 매번 계산) | 풀이 2 (Prefix + 전역 사전 계산) |
|------|---------|--------|--------|
| 시간 복잡도 | $O((n+m) \log n)$ | $O(n + m)$ | $O(n + m)$ (pow10 계산은 전역에서 1회) |
| 공간 복잡도 | $O(n)$ (트리 노드) | $O(n)$ | $O(n + \text{MAX\_N})$ |
| 핵심 기법 | 구간을 재귀적으로 병합하는 세그먼트 트리 | prefix 배열로 병합 규칙을 직접 계산 | prefix 배열 + pow10 전역 캐싱 |

문자열이 고정되고 쿼리만 여러 번 들어오는 이 문제의 특성상, 트리 구조 없이도 병합 규칙을 prefix 형태로 표현할 수 있다는 것이 핵심이다. 따라서 세그먼트 트리보다는 prefix 배열 방식(풀이 1, 풀이 2)이 구현이 간단하고 쿼리당 $O(1)$로 더 빠르다. 여러 테스트 케이스에 걸쳐 반복 호출되는 채점 환경까지 고려한다면 풀이 2처럼 `pow10`을 전역으로 한 번만 계산해두는 최적화가 유효하다.
