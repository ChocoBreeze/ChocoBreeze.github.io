# JDK 종류와 실무 사용 시 고려사항

## 1. JDK란 무엇인가

JDK는 <strong>Java Development Kit</strong>의 약자로, Java 애플리케이션을 개발하고 실행하기 위해 필요한 도구 모음이다. Java로 작성한 소스 코드는 바로 실행되는 것이 아니라, 먼저 컴파일 과정을 거쳐 JVM이 이해할 수 있는 바이트코드 형태로 변환된다. 이때 Java 소스 코드를 컴파일하는 도구가 `javac`이며, 컴파일된 프로그램을 실행하는 도구가 `java`이다. 이러한 개발 및 실행 도구들이 포함된 패키지가 JDK이다.

Java 실행 환경을 이해하기 위해서는 JVM, JRE, JDK의 차이를 구분할 필요가 있다. <strong>JVM(Java Virtual Machine)</strong>은 Java 바이트코드를 실제로 실행하는 가상 머신이다. <strong>JRE(Java Runtime Environment)</strong>는 JVM과 Java 프로그램 실행에 필요한 기본 라이브러리를 포함한 실행 환경이다. 반면 <strong>JDK(Java Development Kit)</strong>는 JRE에 더해 컴파일러, 디버깅 도구, 패키징 도구 등 개발에 필요한 도구까지 포함한다.

따라서 단순히 Java 프로그램을 실행하기만 한다면 JRE만으로도 가능하지만, Java 프로그램을 직접 작성하고 컴파일하려면 JDK가 필요하다. 개발자는 일반적으로 JRE가 아니라 JDK를 설치해야 하며, IntelliJ, Eclipse, Maven, Gradle과 같은 개발 도구들도 내부적으로 JDK를 사용한다.

정리하면 JDK, JRE, JVM의 관계는 다음과 같다.

| 구분  | 의미                       | 역할                 |
| --- | ------------------------ | ------------------ |
| JVM | Java Virtual Machine     | Java 바이트코드 실행      |
| JRE | Java Runtime Environment | Java 프로그램 실행 환경    |
| JDK | Java Development Kit     | Java 개발 및 실행 도구 모음 |

즉, JDK는 Java 개발 환경의 가장 기본이 되는 구성 요소이며, Java 프로젝트를 빌드하고 실행하기 위해 반드시 올바르게 설치 및 설정되어야 한다.

## 2. JDK가 여러 배포판으로 나뉘게 된 배경

초기 Java는 Sun Microsystems가 주도하여 개발한 기술이었다. 당시에는 Java를 개발하기 위해 Sun에서 제공하는 JDK를 설치하는 방식이 일반적이었고, JDK 배포판을 별도로 선택해야 한다는 인식은 크지 않았다. 즉, Java 개발 환경은 비교적 단순하게 이해될 수 있었다.

그러나 Java가 기업 시스템, 서버 애플리케이션, 금융, 공공, 제조, Android 등 다양한 분야에서 널리 사용되면서 상황이 달라졌다. Java는 단순한 프로그래밍 언어를 넘어 여러 산업에서 공통으로 사용하는 핵심 인프라가 되었고, 이에 따라 특정 회사의 제품에만 의존하는 구조는 기업 입장에서 부담이 될 수 있었다.

이러한 배경에서 Java의 오픈소스 구현체인 <strong>OpenJDK</strong>가 중요해졌다. OpenJDK는 Java를 구현한 공개 소스 코드 기반 프로젝트로, 여러 기업과 커뮤니티가 함께 참여할 수 있는 구조를 제공한다. 이후 Oracle, Eclipse, Amazon, Microsoft, Azul, Red Hat 등 여러 주체가 OpenJDK를 기반으로 각자의 JDK 배포판을 빌드하고 제공하게 되었다.

즉, 현재 JDK가 여러 종류로 나뉘어 보이는 이유는 Java 자체가 여러 개이기 때문이 아니라, 동일한 Java 표준과 OpenJDK 기반을 바탕으로 여러 회사와 재단이 각자의 방식으로 빌드, 테스트, 배포, 지원을 제공하기 때문이다.

이 구조는 다음과 같이 이해할 수 있다.

```text
Java 표준 및 OpenJDK 소스 코드
        ↓
여러 회사와 재단이 빌드 및 배포
        ↓
Oracle JDK
Oracle OpenJDK
Eclipse Temurin
Amazon Corretto
Microsoft OpenJDK
Azul Zulu
Red Hat OpenJDK
...
```

따라서 JDK를 선택할 때는 단순히 "Java를 설치한다"는 관점만으로는 부족하다. 어떤 Java 버전을 사용할 것인지, 어떤 배포판을 사용할 것인지, 해당 배포판의 라이선스와 지원 정책은 어떤지 함께 검토해야 한다. 특히 회사 환경에서는 Oracle JDK의 라이선스 정책, OpenJDK 계열 배포판의 사용 가능 여부, 운영 서버와 개발 PC의 JDK 버전 일치 여부가 중요한 검토 대상이 된다.

## 3. OpenJDK와 Oracle JDK의 관계

JDK 배포판을 이해할 때 가장 헷갈리기 쉬운 부분은 <strong>OpenJDK</strong>, <strong>Oracle OpenJDK</strong>, <strong>Oracle JDK</strong>의 관계이다. 이름이 비슷하지만 각각의 의미는 다르다.

<strong>OpenJDK</strong>는 Java의 오픈소스 구현 프로젝트이다. Java 표준을 실제로 구현한 공개 소스 코드 기반 프로젝트라고 볼 수 있으며, 현재 여러 JDK 배포판의 공통 기반이 된다. 즉, OpenJDK는 특정 회사의 단일 제품이라기보다 Java 생태계의 중심이 되는 오픈소스 기반이라고 이해하는 것이 좋다.

<strong>Oracle JDK</strong>는 Oracle이 제공하는 JDK 배포판이다. Oracle은 Java 생태계에서 중요한 역할을 하는 기업이며, Oracle JDK는 Oracle이 빌드하고 배포하는 공식 JDK 제품이다. 과거에는 Oracle JDK가 Java 개발 환경에서 가장 대표적인 선택지로 사용되었지만, 기업 환경에서는 라이선스와 지원 정책을 반드시 확인해야 한다.

반면 <strong>Oracle OpenJDK</strong>는 Oracle이 OpenJDK 소스를 기반으로 빌드하여 제공하는 OpenJDK 배포판이다. 즉, Oracle JDK와 Oracle OpenJDK는 모두 Oracle이 제공하지만, 배포 목적과 라이선스, 지원 정책에서 차이가 있다.

이를 정리하면 다음과 같다.

| 구분             | 의미                      | 특징                         |
| -------------- | ----------------------- | -------------------------- |
| OpenJDK        | Java의 오픈소스 구현 프로젝트      | 여러 JDK 배포판의 공통 기반          |
| Oracle OpenJDK | Oracle이 제공하는 OpenJDK 빌드 | OpenJDK 기반 무료 배포판          |
| Oracle JDK     | Oracle이 제공하는 공식 JDK 제품  | 기업 사용 시 라이선스 및 지원 정책 확인 필요 |

현재 대부분의 주요 JDK 배포판은 OpenJDK를 기반으로 한다. 예를 들어 Eclipse Temurin, Amazon Corretto, Microsoft OpenJDK, Azul Zulu, Red Hat OpenJDK 등은 모두 OpenJDK 기반 배포판으로 볼 수 있다. 따라서 일반적인 Java 문법이나 애플리케이션 동작 방식은 배포판마다 크게 달라지지 않는다.

다만 실무에서는 기능 차이보다 <strong>라이선스, 지원 기간, 보안 업데이트 제공 방식, 회사 표준 여부</strong>가 더 중요하다. 같은 Java 17이라 하더라도 Oracle JDK 17을 사용하는지, Temurin JDK 17을 사용하는지, Amazon Corretto 17을 사용하는지에 따라 회사의 라이선스 검토 대상이나 운영 정책이 달라질 수 있다.

따라서 JDK를 선택할 때는 단순히 "Java 17을 사용한다"라고만 표현하기보다, 가능하면 다음과 같이 버전과 배포판을 함께 명시하는 것이 좋다.

```text
Oracle JDK 17
Oracle OpenJDK 17
Eclipse Temurin JDK 17
Amazon Corretto 17
Microsoft OpenJDK 17
```

결론적으로 OpenJDK는 여러 JDK 배포판의 기반이 되는 오픈소스 프로젝트이고, Oracle JDK는 Oracle이 제공하는 JDK 제품이다. Oracle OpenJDK는 그 사이에 위치한 개념으로, Oracle이 OpenJDK를 기반으로 빌드하여 배포하는 OpenJDK 계열 JDK라고 이해할 수 있다.

## 4. 주요 JDK 배포판

현재 Java 개발 환경에서는 하나의 JDK만 사용되는 것이 아니라, 여러 회사와 재단이 제공하는 다양한 JDK 배포판이 사용된다. 대부분의 주요 JDK 배포판은 OpenJDK를 기반으로 하지만, 제공 주체, 라이선스, 지원 정책, 업데이트 방식, 사용 환경에 따라 차이가 있다.

JDK 배포판을 선택할 때는 단순히 "Java가 실행되는가"만 볼 것이 아니라, 회사에서 사용 가능한 배포판인지, 보안 업데이트가 지속적으로 제공되는지, 운영 환경과 호환되는지 함께 확인해야 한다.

대표적인 JDK 배포판은 다음과 같다.

| 배포판                        | 제공 주체            | 특징                                                                                                          |
| -------------------------- | ---------------- | ----------------------------------------------------------------------------------------------------------- |
| Oracle JDK                 | Oracle           | Oracle이 제공하는 공식 JDK 제품이다. 기업 환경에서는 라이선스 및 지원 정책을 확인해야 한다.                                                   |
| Oracle OpenJDK             | Oracle           | Oracle이 OpenJDK를 기반으로 빌드하여 제공하는 배포판이다. Oracle JDK와 이름이 비슷하지만 성격과 정책이 다르다.                                   |
| Eclipse Temurin            | Eclipse Adoptium | OpenJDK 기반의 무료 JDK 배포판이다. 개인 개발, 회사 개발 PC, 서버 환경에서 무난하게 사용할 수 있는 선택지로 많이 사용된다.                              |
| Amazon Corretto            | Amazon           | Amazon이 제공하는 OpenJDK 배포판이다. AWS 환경에서 많이 사용되며, 장기 지원을 제공하는 배포판으로 알려져 있다.                                     |
| Microsoft Build of OpenJDK | Microsoft        | Microsoft가 제공하는 OpenJDK 배포판이다. Windows 및 Azure 환경에서 사용하기 좋은 선택지이다.                                          |
| Azul Zulu                  | Azul             | Azul에서 제공하는 OpenJDK 배포판이다. 다양한 플랫폼 지원과 기업용 장기 지원 옵션이 특징이다.                                                  |
| Red Hat OpenJDK            | Red Hat          | Red Hat이 제공하는 OpenJDK 배포판이다. RHEL 등 Red Hat 계열 기업 리눅스 환경에서 자주 사용된다.                                         |
| GraalVM                    | Oracle           | 일반적인 JDK 기능뿐 아니라 Native Image, 고성능 런타임 등 특수 기능을 제공한다. 일반 Java 개발보다는 성능 최적화나 네이티브 실행 파일 생성이 필요한 경우에 주로 검토된다. |

이 중 일반적인 개발 환경에서 많이 선택되는 배포판은 <strong>Eclipse Temurin</strong>, <strong>Amazon Corretto</strong>, <strong>Microsoft OpenJDK</strong> 등이다. 이들은 모두 OpenJDK 기반이므로 일반적인 Java 문법, Spring 애플리케이션, Maven/Gradle 빌드, IDE 개발 환경에서 큰 차이 없이 사용할 수 있다.

반면 <strong>Oracle JDK</strong>는 Oracle이 제공하는 공식 JDK라는 장점이 있지만, 회사 환경에서는 라이선스 정책을 반드시 확인해야 한다. 따라서 회사 표준이 정해져 있지 않은 경우에는 Oracle JDK보다 OpenJDK 계열 배포판을 우선 검토하는 경우가 많다.

JDK 배포판을 이해할 때 중요한 점은, 배포판이 다르다고 해서 Java 언어 자체가 달라지는 것은 아니라는 점이다. 예를 들어 같은 Java 17 기반이라면 Oracle JDK 17, Temurin JDK 17, Amazon Corretto 17은 대부분의 일반적인 Java 애플리케이션에서 동일하게 동작한다. 차이는 주로 빌드 제공 주체, 업데이트 정책, 지원 기간, 라이선스, 운영 환경 적합성에서 발생한다.

따라서 JDK를 명시할 때는 다음과 같이 버전과 배포판을 함께 적는 것이 좋다.

```text
Temurin JDK 17
Amazon Corretto 17
Oracle JDK 21
Microsoft OpenJDK 21
```

이처럼 JDK 선택은 "어떤 Java 버전을 사용할 것인가"와 "어떤 배포판을 사용할 것인가"를 함께 결정하는 문제이다. 실무에서는 프로젝트 요구 버전, 회사 표준, 운영 서버 환경, 라이선스 정책을 함께 고려하여 JDK 배포판을 선택해야 한다.

## 5. JDK 사용 시 주요 이슈

JDK는 Java 개발과 실행의 기본 환경이지만, 실제 프로젝트에서는 단순히 JDK를 설치하는 것만으로 끝나지 않는다. JDK 배포판, Java 버전, 환경변수, IDE 설정, 빌드 도구 설정, 운영 서버 환경이 서로 맞지 않으면 다양한 문제가 발생할 수 있다. 특히 회사 환경에서는 라이선스, 보안 업데이트, 운영 표준까지 함께 고려해야 한다.

### 5.1 Oracle JDK 라이선스 이슈

JDK 관련 이슈 중 가장 대표적인 것은 Oracle JDK 라이선스 문제이다. Oracle JDK는 Oracle이 제공하는 공식 JDK 제품이지만, 기업 환경에서 사용할 경우 버전과 사용 조건에 따라 라이선스 검토가 필요할 수 있다.

개인 학습이나 테스트 환경에서는 크게 문제되지 않을 수 있지만, 회사 개발 PC나 운영 서버에서 Oracle JDK를 사용하는 경우에는 회사의 Oracle Java 사용 정책이나 계약 여부를 확인해야 한다. 라이선스 검토 없이 Oracle JDK를 사용하면 추후 비용이나 감사 이슈가 발생할 수 있다.

이 때문에 많은 회사에서는 Oracle JDK 대신 Eclipse Temurin, Amazon Corretto, Microsoft OpenJDK, Azul Zulu와 같은 OpenJDK 계열 배포판을 사용한다. 이러한 배포판은 일반적인 Java 개발과 실행에 충분히 사용할 수 있으며, 라이선스 부담을 줄일 수 있는 선택지로 많이 검토된다.

### 5.2 Java 버전 불일치 이슈

Java는 버전에 따라 지원하는 문법, 라이브러리, class file 버전이 다르다. 따라서 프로젝트가 요구하는 Java 버전과 실제 개발 PC 또는 운영 서버에서 사용하는 Java 버전이 다르면 문제가 발생할 수 있다.

예를 들어 Java 17로 컴파일한 프로그램을 Java 8 환경에서 실행하면 class file 버전이 맞지 않아 실행 오류가 발생할 수 있다. 반대로 프로젝트는 Java 8 기준인데 개발 PC에서 Java 21 문법을 사용하면, 운영 서버나 다른 개발자의 환경에서 빌드가 실패할 수 있다.

대표적인 오류 메시지는 다음과 같다.

```text
Unsupported class file major version
Unsupported major.minor version
class file has wrong version
release version 17 not supported
```

따라서 프로젝트에서는 사용 Java 버전을 명확히 정해야 하며, 개발 PC, IDE, Maven/Gradle, 운영 서버가 동일한 major 버전을 사용하도록 관리해야 한다.

### 5.3 JDK 배포판 차이에 따른 이슈

대부분의 JDK 배포판은 OpenJDK 기반이므로 일반적인 Java 애플리케이션에서는 큰 차이 없이 동작한다. 그러나 배포판에 따라 업데이트 정책, 지원 기간, 플랫폼 지원, 기업 지원 옵션, 라이선스 조건이 다를 수 있다.

예를 들어 같은 Java 17이라도 다음과 같이 여러 배포판이 존재할 수 있다.

```text
Oracle JDK 17
Eclipse Temurin JDK 17
Amazon Corretto 17
Microsoft OpenJDK 17
Azul Zulu 17
```

일반 개발에서는 큰 차이를 느끼지 못할 수 있지만, 회사에서는 어떤 배포판을 표준으로 사용할지 정해두는 것이 좋다. 개발자는 같은 Java 버전을 사용한다고 생각해도 실제로는 서로 다른 배포판을 사용하고 있을 수 있기 때문이다.

### 5.4 JAVA_HOME 및 PATH 설정 이슈

JDK 설치 후 가장 자주 발생하는 문제는 환경변수 설정 오류이다. Windows 환경에서는 `JAVA_HOME`과 `PATH` 설정이 올바르지 않으면 의도한 JDK가 아니라 다른 Java가 실행될 수 있다.

예를 들어 JDK 17을 설치했는데 터미널에서 `java --version`을 실행하면 Java 8이 나오는 경우가 있다. 이는 기존에 설치된 Java 경로가 PATH에서 더 앞에 위치하기 때문이다.

확인은 다음 명령어로 할 수 있다.

```powershell
java --version
javac --version
where java
where javac
echo $env:JAVA_HOME
```

여기서 `java`와 `javac`가 같은 JDK 경로를 바라보고 있는지 확인해야 한다. 또한 `JAVA_HOME`이 실제 설치된 JDK 폴더를 가리키는지, `PATH`에 `%JAVA_HOME%\bin`이 올바르게 등록되어 있는지도 확인해야 한다.

### 5.5 IDE에서 사용하는 JDK 불일치 이슈

IntelliJ나 Eclipse 같은 IDE는 시스템 환경변수와 별도로 프로젝트 JDK를 설정할 수 있다. 따라서 터미널에서는 Java 17을 사용하고 있어도, IDE에서는 Java 8이나 Java 11을 사용하고 있을 수 있다.

IntelliJ에서는 다음 항목을 확인해야 한다.

```text
File > Project Structure > Project SDK
Settings > Build Tools > Gradle > Gradle JVM
Settings > Build Tools > Maven > JDK for importer
```

Eclipse에서는 다음 항목을 확인해야 한다.

```text
Window > Preferences > Java > Installed JREs
Project > Properties > Java Build Path
Project > Properties > Java Compiler
```

특히 Eclipse 프로젝트에서는 `.classpath`, `.project` 파일에 설정된 JRE/JDK 정보와 실제 설치된 JDK가 맞지 않으면 빌드 오류가 발생할 수 있다.

### 5.6 Maven 및 Gradle 빌드 JDK 이슈

Maven과 Gradle은 IDE와 별도로 JDK를 사용할 수 있다. 따라서 IDE에서는 정상적으로 실행되지만, 터미널에서 `mvn test`나 `gradle build`를 실행하면 실패하는 경우가 있다.

Maven이 사용하는 Java 버전은 다음 명령어로 확인할 수 있다.

```bash
mvn -version
```

Gradle이 사용하는 Java 버전은 다음 명령어로 확인할 수 있다.

```bash
gradle -version
```

프로젝트의 `pom.xml`이나 `build.gradle`에서도 Java 버전을 지정할 수 있다.

Maven 예시는 다음과 같다.

```xml
<properties>
    <maven.compiler.source>17</maven.compiler.source>
    <maven.compiler.target>17</maven.compiler.target>
</properties>
```

Gradle 예시는 다음과 같다.

```gradle
java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(17)
    }
}
```

이처럼 빌드 도구에서 사용하는 JDK 버전과 프로젝트가 요구하는 Java 버전이 맞아야 안정적으로 빌드할 수 있다.

### 5.7 개발 PC와 운영 서버의 JDK 차이

개발 PC에서는 정상적으로 동작하지만 운영 서버에서 오류가 발생하는 경우도 있다. 대표적인 원인은 개발 PC와 운영 서버의 JDK 버전 또는 배포판이 다르기 때문이다.

예를 들어 개발 PC에서는 Java 17을 사용하고 운영 서버에서는 Java 8을 사용한다면, 개발 PC에서 빌드한 결과물이 운영 서버에서 실행되지 않을 수 있다. 또한 운영 서버의 JDK가 오래된 패치 버전이면 보안 취약점이나 TLS 통신 문제가 발생할 수도 있다.

따라서 운영 환경에서는 다음 항목을 확인해야 한다.

```text
개발 PC와 운영 서버의 Java major 버전이 같은가?
운영 서버의 JDK 패치 버전이 너무 오래되지 않았는가?
운영 서버에서 사용하는 JDK 배포판이 회사 표준에 맞는가?
빌드 결과물이 운영 서버의 Java 버전과 호환되는가?
```

### 5.8 보안 업데이트 및 LTS 이슈

JDK도 지속적인 보안 업데이트가 필요하다. 오래된 JDK를 계속 사용하면 보안 취약점이나 호환성 문제가 발생할 수 있다. 특히 서버 환경에서는 단순히 Java major 버전만 보는 것이 아니라, patch 버전까지 관리해야 한다.

예를 들어 같은 Java 17이라도 다음과 같이 패치 버전이 다를 수 있다.

```text
Java 17.0.8
Java 17.0.10
Java 17.0.13
```

회사에서는 LTS 버전을 중심으로 사용하고, 정기적으로 보안 업데이트를 적용하는 정책을 갖는 것이 좋다. 일반적으로 Java 8, 11, 17, 21과 같은 LTS 버전이 많이 사용되며, 신규 프로젝트에서는 비교적 최신 LTS 버전을 검토할 수 있다.

### 5.9 회사 보안 인증서 및 TLS 이슈

회사 네트워크 환경에서는 보안 장비나 프록시 때문에 Java 애플리케이션에서 HTTPS 통신 오류가 발생할 수 있다. 특히 사내 인증서나 자체 서명 인증서를 사용하는 경우 Java의 truststore에 해당 인증서가 등록되어 있지 않으면 SSL 오류가 발생한다.

대표적인 오류는 다음과 같다.

```text
PKIX path building failed
unable to find valid certification path
javax.net.ssl.SSLHandshakeException
```

이 경우에는 회사 인증서를 Java truststore에 등록하거나, 별도의 truststore를 지정해야 할 수 있다. Python에서 인증서 파일을 `verify` 옵션으로 지정하는 것처럼, Java에서는 `cacerts` 또는 `-Djavax.net.ssl.trustStore` 설정을 사용하는 방식으로 처리한다.

### 5.10 정리

JDK 관련 문제는 대부분 하나의 원인만으로 발생하지 않는다. JDK 버전, 배포판, 환경변수, IDE 설정, 빌드 도구, 운영 서버 환경이 서로 맞지 않을 때 문제가 발생한다. 따라서 JDK 이슈를 점검할 때는 다음 순서로 확인하는 것이 좋다.

```text
1. 회사에서 허용된 JDK 배포판인가?
2. 프로젝트가 요구하는 Java 버전은 무엇인가?
3. java와 javac가 같은 JDK를 가리키는가?
4. JAVA_HOME과 PATH가 올바르게 설정되어 있는가?
5. IDE의 Project SDK가 프로젝트 기준과 일치하는가?
6. Maven 또는 Gradle이 사용하는 JDK가 맞는가?
7. 개발 PC와 운영 서버의 JDK 버전이 호환되는가?
8. 보안 업데이트와 LTS 지원 정책이 적절한가?
9. 회사 인증서나 TLS 관련 문제가 없는가?
```

결국 JDK 사용 시 가장 중요한 것은 "설치되어 있는가"가 아니라, 프로젝트와 회사 환경에 맞는 JDK가 일관되게 사용되고 있는가이다.

## 6. JDK 선택 기준

JDK를 선택할 때는 단순히 최신 버전을 설치하는 것보다, 프로젝트 요구사항과 회사 운영 환경을 함께 고려해야 한다. JDK는 Java 애플리케이션의 개발, 빌드, 실행에 직접 영향을 주기 때문에 버전이나 배포판이 맞지 않으면 개발 PC에서는 정상 동작하더라도 운영 서버에서 문제가 발생할 수 있다.

JDK 선택 시 가장 먼저 확인해야 할 것은 <strong>프로젝트가 요구하는 Java 버전</strong>이다. 기존 프로젝트는 Java 8, Java 11, Java 17과 같이 특정 버전에 맞춰 개발되어 있을 수 있다. 이 경우 개발자가 임의로 최신 JDK를 사용하면 문법 호환성, 라이브러리 호환성, class file 버전 문제 등이 발생할 수 있다. 따라서 신규 프로젝트가 아니라면 기존 프로젝트의 `pom.xml`, `build.gradle`, IDE 설정, 운영 서버 Java 버전을 먼저 확인해야 한다.

두 번째로 확인해야 할 것은 <strong>JDK 배포판</strong>이다. 같은 Java 17이라도 Oracle JDK 17, Eclipse Temurin JDK 17, Amazon Corretto 17, Microsoft OpenJDK 17 등 여러 배포판이 존재한다. 대부분 OpenJDK 기반이기 때문에 일반적인 Java 애플리케이션에서는 큰 차이 없이 동작하지만, 회사 환경에서는 라이선스, 보안 업데이트, 장기 지원 정책이 중요하다.

특히 Oracle JDK는 기업 환경에서 라이선스 검토가 필요할 수 있으므로, 회사의 Java 사용 정책이나 Oracle 계약 여부를 확인해야 한다. 별도의 회사 표준이 없다면 Eclipse Temurin, Amazon Corretto, Microsoft OpenJDK, Azul Zulu와 같은 OpenJDK 계열 배포판을 우선 검토할 수 있다.

세 번째로는 <strong>개발 PC와 운영 서버의 일관성</strong>을 확인해야 한다. 개발자는 Java 21을 사용하고 운영 서버는 Java 8을 사용한다면, 개발 PC에서 빌드한 결과물이 운영 서버에서 실행되지 않을 수 있다. 따라서 개발, 빌드, 테스트, 운영 환경에서 사용하는 Java major 버전은 최대한 일치시키는 것이 좋다.

JDK 선택 기준은 다음과 같이 정리할 수 있다.

| 상황            | 권장 기준                                                |
| ------------- | ---------------------------------------------------- |
| 개인 학습 및 일반 개발 | Eclipse Temurin JDK의 LTS 버전 사용                       |
| 회사 개발 PC      | 회사 표준 JDK 우선 적용                                      |
| 회사 표준이 없는 경우  | Temurin, Corretto, Microsoft OpenJDK 등 OpenJDK 계열 검토 |
| Oracle JDK 사용 | 회사 라이선스 정책 및 계약 여부 확인                                |
| 레거시 프로젝트      | 프로젝트가 요구하는 Java 버전 우선 사용                             |
| 신규 프로젝트       | 최신 LTS 버전 검토                                         |
| 운영 서버         | 개발 및 빌드 환경과 Java major 버전 일치                         |
| 보안이 중요한 환경    | LTS 및 정기 보안 업데이트 제공 여부 확인                            |

실무에서는 다음과 같은 순서로 판단하는 것이 좋다.

```text
1. 프로젝트가 요구하는 Java 버전을 확인한다.
2. 회사에서 허용하거나 표준으로 정한 JDK 배포판이 있는지 확인한다.
3. Oracle JDK를 사용할 경우 라이선스 정책을 확인한다.
4. 회사 표준이 없다면 OpenJDK 계열 배포판을 우선 검토한다.
5. 개발 PC, IDE, Maven/Gradle, 운영 서버의 JDK 버전이 일치하는지 확인한다.
6. LTS 버전과 보안 업데이트 제공 여부를 확인한다.
```

결론적으로 JDK 선택은 "어떤 Java를 설치할 것인가"의 문제가 아니라, <strong>프로젝트와 조직 환경에 맞는 Java 개발·실행 환경을 일관되게 구성하는 문제</strong>이다. Java 버전, JDK 배포판, 라이선스, 지원 정책, 운영 서버 호환성을 함께 고려해야 하며, 이를 명확히 관리해야 안정적인 개발과 운영이 가능하다.

따라서 특별한 회사 표준이나 Oracle JDK 사용 사유가 없다면, 일반적인 개발 환경에서는 Eclipse Temurin, Amazon Corretto, Microsoft OpenJDK와 같은 OpenJDK 계열 LTS 버전을 사용하는 것이 무난하다. 반대로 기존 프로젝트나 운영 서버가 특정 Java 버전을 요구한다면, 최신 버전보다 프로젝트 기준 버전을 우선해야 한다.
