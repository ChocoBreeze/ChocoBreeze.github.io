> MCP Inspector를 켤 때마다 `http://localhost:6274/?MCP_PROXY_AUTH_TOKEN=...`처럼 URL이 매번 바뀌는 걸 보고 "이건 뭐고 왜 이렇게 하는가"를 정리한 글.

---

## 겪은 현상

`uv run mcp dev mcp_server.py`를 실행할 때마다 터미널에 매번 다른 URL이 출력된다.

```
http://localhost:6274/?MCP_PROXY_AUTH_TOKEN=a1b2c3d4e5f6...
http://localhost:6274/?MCP_PROXY_AUTH_TOKEN=9f8e7d6c5b4a...   ← 재시작하면 값이 바뀜
```

포트(6274)는 그대로인데 뒤에 붙는 토큰 값만 매번 랜덤하게 바뀐다. 즐겨찾기에 저장해둔 예전 링크로 접속하면 인증 실패가 뜬다.

---

## 정체: Capability URL (토큰-인-URL 방식)

이 패턴의 정식 이름은 **Capability URL**(또는 token-in-URL)이다. 핵심 아이디어는 한 줄로 요약된다.

> **"이 URL 전체를 아는 것 자체가 곧 접근 권한을 갖는 것이다."**

아이디/비밀번호를 입력하는 로그인 화면이 따로 없다. URL 안에 이미 열쇠(토큰)가 박혀 있고, 그 열쇠를 제시하는 것만으로 통과된다.

## 동작 방식

```
1. mcp dev 실행
   ↓
2. 프록시 서버가 랜덤 문자열(토큰) 생성
   ↓
3. 그 문자열을 서버 메모리에 "유효한 토큰"으로 저장
   ↓
4. 터미널에 토큰이 붙은 URL 출력
   ↓
5. 브라우저가 그 URL로 요청 전송 (토큰이 쿼리 파라미터로 실려감)
   ↓
6. 서버가 요청에 실린 토큰과 자신이 저장해둔 토큰을 비교
   → 일치하면 통과, 불일치하면 거부
```

로그인 세션처럼 "누구인지"를 구별하는 게 아니라, **"허가받은 요청인지 아닌지"만 이진 판정**하는 매우 단순한 검증이다.

## 왜 아이디/비밀번호 대신 이 방식을 쓰는가

로컬 개발 도구에서 회원가입·로그인 시스템을 따로 만들 이유가 없다. 대신 이렇게 추론한다:

- **터미널에 그 URL을 출력한 사람 = 이 컴퓨터에서 해당 명령을 실행할 권한이 있는 사람** (이미 검증된 것으로 간주)
- 그 URL을 브라우저에 붙여넣는 행위 자체가 "나는 그 터미널을 봤다"는 증명이 된다

그래서 무거운 인증 절차 없이, **"URL을 아는가"**라는 최소 조건만으로 충분하다.

## 비유

- **일반 로그인(ID/PW)** = 은행 계좌. 신원을 정확히 확인해야 열람 가능.
- **Capability URL** = **호텔 카드키**. "당신이 누구인지"는 묻지 않는다. "이 카드를 갖고 있는가"만 본다. 카드를 가진 사람 = 그 방에 들어갈 자격이 있는 사람으로 취급된다.

## 실전에서 만나는 곳

이 패턴은 생각보다 자주 쓰인다.

- **Google Docs/Drive의 "링크 아는 모든 사용자"** — 링크 자체에 접근 권한이 인코딩되어 있어, 로그인 없이 그 링크만으로 바로 열람 가능
- **AWS S3 presigned URL** — 일정 시간만 유효한 서명이 URL에 포함된 다운로드 링크
- **이메일의 "구독 취소" 링크** — 로그인 없이 클릭 한 번으로 처리
- **MCP Inspector** — 지금 이 사례

## 한계

편리하지만 대가가 있다. **URL 자체가 곧 비밀번호**라서, 그 URL이 다음과 같은 경로로 새어나가면 토큰을 아는 모든 사람이 접근 가능해진다.

- 브라우저 히스토리
- 서버/프록시 로그 파일 (쿼리 파라미터가 그대로 로그에 남는 경우가 많다)
- 화면 캡처, 화면 공유
- Slack/이메일 등으로 링크를 그대로 전달

그래서 이 방식은 보통 다음 조건과 함께 쓰인다:

- **`localhost` 전용** — 애초에 외부 네트워크에서 접근 불가 (MCP Inspector가 이 경우)
- **짧은 유효 시간** — presigned URL처럼 몇 분~몇 시간 내 자동 만료
- **1회성 토큰** — 재시작할 때마다 새 토큰 발급 (Inspector가 매번 다른 토큰을 주는 이유)

## 그래서 실무에서는

- 재시작할 때마다 **터미널에 새로 출력되는 URL을 그대로 다시 열어야** 한다. 즐겨찾기해둔 예전 URL은 토큰이 달라 통하지 않는다.
- 브라우저 탭을 재사용하지 말고, 매번 터미널에 찍힌 새 링크를 여는 습관을 들이는 편이 편하다.
- 이런 URL을 캡처해서 공유할 때는 토큰 부분을 가리는 게 안전하다 — 카드키 사진을 공개로 올리는 것과 같기 때문이다.

---

## 정리

> Capability URL = **"URL 전체가 곧 열쇠"**인 인증 방식. 아이디/비밀번호로 "누구인지"를 확인하는 대신, 정확한 URL을 아는지 여부로 "권한이 있는지"만 확인한다. 편리하지만 URL이 새면 곧 토큰이 새는 것과 같으므로, 로컬 전용·짧은 유효시간·1회성 발급과 함께 쓰이는 것이 일반적이다.

MCP Inspector가 재시작마다 토큰을 새로 발급하는 것도, 결국 "카드키를 매번 새로 찍어내는" 것과 같은 이유다.
